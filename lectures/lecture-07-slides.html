6.1 — User-defined Function Basics

Why Functions? The Problem with Repeated Code

❌ Without functions — cluttered and error-prone:

# Fahrenheit → Celsius, written 3 times!
c1 = (f1 - 32.0) * (5.0 / 9.0)
c2 = (f2 - 32.0) * (5.0 / 9.0)
c3 = (f3 + 32.0) * (5.0 / 9.0)  # ← BUG!
The third line uses + 32.0 instead of - 32.0. Copy-pasting repeated code creates more opportunities for subtle bugs.

✅ With a function — clean, safe, written once:

def F2C(f):
    c = (f - 32.0) * (5.0 / 9.0)
    return c

c1 = F2C(f1)  # call it —
c2 = F2C(f2)  # as many —
c3 = F2C(f3)  # times as needed
The calculation is written once. Fix a bug once → fixed everywhere it's called.
Definition

A function is a named series of statements. Functions reduce redundancy, prevent errors, and keep programs readable — even without repeated code.

6.1 — User-defined Function Basics

Anatomy of a Python Function

def calc_pizza_area(pizza_diameter):        # ← DEFINITION  (def keyword + name + parameter(s))
    """Returns area of a pizza in sq inches."""  # ← DOCSTRING  (first line of body)
    pi_val       = 3.14159265               # ← LOCAL VARIABLE
    pizza_radius = pizza_diameter / 2.0
    pizza_area   = pi_val * pizza_radius * pizza_radius
    return pizza_area                       # ← RETURN STATEMENT

# ── calling the function ──────────────────────────────────────────────────────
area   = calc_pizza_area(12.0)              # ← CALL (argument 12.0 → parameter pizza_diameter)
print(f"12-inch pizza: {area:.3f} sq in")  # → 12-inch pizza: 113.097 sq in
  • def — keyword that starts a function definition
  • Name — lowercase_with_underscores (calc_pizza_area). Use verbs.
  • Parameters — named inputs listed in ( )
  • Body — indented block; runs when the function is called
  • return — sends a value back to the caller
  • Function call — uses the name + arguments in ( )
  • Built-ins you know: input(), int(), len(), print()…
  • Define first, call second — Python reads top-to-bottom
6.1 — Return Statements & Parameters

Return Statements & Parameters vs. Arguments

import math

# One parameter, returns a value
def compute_square(num_to_square):
    return num_to_square * num_to_square

# Multiple parameters
def calc_pizza_volume(pizza_diameter, pizza_height):
    radius = pizza_diameter / 2.0
    area   = math.pi * radius * radius
    return area * pizza_height

# Storing the returned value
sq = compute_square(7)        # sq = 49
print(f"7 squared is {sq}")

# Using call directly in an expression
print(compute_square(5) + compute_square(3))  # → 34

# Multiple arguments — matched left to right
vol = calc_pizza_volume(12.0, 0.3)
print(f"Volume: {vol:.3f} cubic in")
Parameter vs. Argument

Parameter — the variable name in the definition (the placeholder).

Argument — the actual value supplied in the call.

  • Arguments map left → right to parameters
  • An argument can be a literal, variable, or expression
  • No parameters? Still need the empty ( )
  • A function call evaluates to its returned value — usable in any expression
No return statement (or bare return) → function returns None. A function returning None cannot be used in a math expression.
Invalid parameter:
def my_fct(x + 5): → SyntaxError. Parameters must be plain names.
6.1 — Hierarchical Function Calls

Functions Calling Functions

import math

def calc_circle_area(diameter):
    """Returns the area of a circle."""
    radius = diameter / 2.0
    return math.pi * radius * radius

def calc_pizza_calories(diameter):
    """Returns total calories for a pepperoni pizza."""
    cal_per_sq_inch = 16.7
    return calc_circle_area(diameter) * cal_per_sq_inch

# Main program — reads like plain English
print(f"12-inch pizza: {calc_pizza_calories(12.0):.0f} cal")
print(f"14-inch pizza: {calc_pizza_calories(14.0):.0f} cal")

# You have been doing this ALL semester already:
user_val = int(input("Enter a number: "))
#           ^inner   ^outer
# input() runs first → result passed to int()
Hierarchical (Nested) Calls

A function's body may call other functions. Complex functions build on simpler ones — each doing exactly one thing.

You've been doing this all semester!
int(input()) is a nested call — input() runs first, its string result is passed to int().
Think in layers: calc_circle_area handles geometry; calc_pizza_calories handles nutrition. Each function has one clear job. The main program orchestrates them.

6.2 — Print Functions

Void Functions: Functions That Print

def print_summary(oid, items, price):
    """Prints a formatted order summary."""
    print(f"Order {oid}:")
    print(f"  Items: {items}")
    print(f"  Total: ${price:.2f}")
    # no return → implicitly returns None

# Call as many times as needed; formatting written once
print_summary(42, 4, 13.99)
print_summary(43, 1, 5.49)

# ── Classic menu system ────────────────────────────────
def print_menu():
    print("Today's Menu:")
    print("  1) Gumbo")
    print("  2) Jambalaya")
    print("  3) Quit\n")

quit_program = False
while not quit_program:
    print_menu()                        # called once per loop
    choice = int(input("Enter choice: "))
    if choice == 3:
        print("Goodbye")
        quit_program = True
    elif choice == 1:
        print("Order: Gumbo")
    elif choice == 2:
        print("Order: Jambalaya")
Void Function

A function with no return statement (or bare return) is void — it performs an action but returns None.

  • Void functions can have any number of parameters
  • Great for formatted output that appears multiple times
  • Adding return 0 makes print_menu() non-void
  • Bare return alone still leaves it void
Key insight: Write complex output once inside the function. Call the function many times. Change the format in one place and it updates everywhere.
A function that both prints and returns a value is not void — but separating those concerns makes code cleaner.
6.3 — Dynamic Typing

Dynamic Typing & Polymorphism

def add(x, y):
    return x + y

# Same function, different argument types:
print(add(5, 7))              # → 12   (int + int)
print(add("Tora", "Bora"))    # → ToraBora  (str + str)
print(add([1, 2], [3, 4]))    # → [1, 2, 3, 4]  (list + list)
print(add(5, "100"))          # → TypeError at runtime!

# * is also polymorphic
print(3 * 4)          # → 12
print("ha" * 3)       # → hahaha

# Dynamic typing — type determined at runtime
num = 5       # references an int
num = "seven" # now references a str — valid in Python!
add(5, "100") → TypeError at runtime. Python checks types when the code runs, not before.
Dynamic Typing

Type is determined when code executes, not when it is written. The same variable can hold different types at different times.

Polymorphism

One operation (+, *, a function) behaves differently based on the types involved. Python's add() function is polymorphic.

Duck Typing

"If it walks, swims, and quacks like a duck — call it a duck." If an object supports the required operation, Python uses it.

Contrast: C, C++, Java use static typing — every variable's type is declared in source code and checked at compile time. More verbose, but catches type errors earlier.
6.4 — Reasons for Defining Functions

Three Reasons to Write Functions

1 · Readability

The main program becomes a short sequence of named function calls that reads like an outline. You understand what the program does without wading through how.

2 · Avoid Redundancy — DRY

Don't Repeat Yourself. Define a function once, call it everywhere. Fix a bug once and it's fixed in every call site.

3 · Modular Development

Break a large program into independent modules. Write and test each function separately, then integrate. Far easier to debug and maintain.

Think of a cruise ship — the bridge, engine room, and galley are separate departments, each with a clear job, working together. Functions are your program's departments.

With functions — the main program:

steps = int(input("Enter steps: "))

feet     = steps_to_feet(steps)
calories = steps_to_calories(steps)

print(f"Feet:     {feet}")
print(f"Calories: {calories}")

Without functions — the main program:

steps = int(input("Enter steps: "))

feet_per_step          = 3
steps_per_minute       = 70.0
cal_per_minute_walking = 3.5

feet     = steps * feet_per_step
minutes  = steps / steps_per_minute
calories = minutes * cal_per_minute_walking

print(f"Feet:     {feet}")
print(f"Calories: {calories}")
Guideline: A function body should generally be 30 lines or fewer. If it's growing beyond that, decompose it further.
6.5 — Writing Mathematical Functions

Mathematical Functions & Calling in Expressions

import math

CM_PER_INCH    = 2.54   # ← UPPER_CASE for constants
INCHES_PER_FT  = 12

def height_US_to_cm(feet, inches):
    """Converts height in feet/inches to centimeters."""
    total_inches = feet * INCHES_PER_FT + inches
    return total_inches * CM_PER_INCH

# A call EVALUATES TO its returned value — use it anywhere
print(height_US_to_cm(6, 4))   # → 193.04

avg_cm = (height_US_to_cm(5, 8) + height_US_to_cm(6, 0)) / 2.0

# Modular: functions calling functions
def calc_circular_base_area(radius):
    return math.pi * radius * radius

def calc_cylinder_volume(base_radius, height):
    return calc_circular_base_area(base_radius) * height

def calc_cylinder_surface_area(base_radius, height):
    return (2 * math.pi * base_radius * height) + \
           (2 * calc_circular_base_area(base_radius))

r = float(input("Enter base radius: "))
h = float(input("Enter height: "))
print(f"Volume:       {calc_cylinder_volume(r, h):.3f}")
print(f"Surface area: {calc_cylinder_surface_area(r, h):.3f}")
calc_circular_base_area() is shared by both volume and surface area. One fix, two functions benefit — modularity in action.
  • Use UPPER_CASE for constants at global scope
  • A function call evaluates to its return value — treat it like a number in any expression
  • 5 + compute_square(4) → 5 + 16 → 21
  • Nested calls: pow(2, pow(3, 2)) → pow(2, 9) → 512
A void function (returns None) cannot be used inside a math expression — it will crash with a TypeError at runtime.
Valid uses of height_US_to_cm():
x = height_US_to_cm(5, 0) ✅
x = 2 * (height_US_to_cm(5, 0) + 1.0) ✅
height_US_to_cm(5, 0) = x ❌ (can't assign to a call)
6.6 — Function Stubs

Function Stubs & Incremental Development

def steps_to_feet(num_steps):
    """Converts steps to feet."""
    feet_per_step = 3
    return num_steps * feet_per_step

# ── STUB style 1: pass — does nothing, returns None ──────
def steps_to_calories(num_steps):
    pass

# ── STUB style 2: print + sentinel return ────────────────
def steps_to_calories(num_steps):
    print("FIXME: finish steps_to_calories")
    return -1       # sentinel: caller knows it's not ready

# ── STUB style 3: hard stop ───────────────────────────────
def steps_to_calories(num_steps):
    raise NotImplementedError

# Main program works immediately (feet part tested now)
steps    = int(input("Enter steps: "))
feet     = steps_to_feet(steps)
calories = steps_to_calories(steps)     # stub — safe placeholder
print(f"Feet: {feet}")
print(f"Calories: {calories}")  # → Calories: None (or -1)
Incremental Development

Write a small amount of code, test it, then add a little more. Don't wait until everything is "done" before running the program.

  • pass — required placeholder when the body would otherwise be empty; returns None
  • return -1 — signals "not ready yet" without crashing
  • raise NotImplementedError — hard stop; prevents downstream code from running on unfinished logic
Stubs are like planning a road trip before you start driving — you know the destinations (functions) before filling in every detail.
6.7 — Functions with Branches / Loops

Functions with Branches & Loops

Normalize a grade to a bounded range:

def normalize_grade(grade):
    upper_bound = 68
    lower_bound = 54
    if grade > upper_bound:
        return upper_bound    # cap at 68
    elif grade <= lower_bound:
        return lower_bound    # floor at 54
    else:
        return grade          # within range

print(normalize_grade(88))   # → 68
print(normalize_grade(5))    # → 54
print(normalize_grade(57))   # → 57
Multiple return statements in different branches is perfectly valid. Execution leaves the function at the first return encountered.

Tiered pricing with if / elif / else:

def calc_ebay_fee(sell_price):
    """eBay fee for books/music/video games."""
    p50      = 0.13   # 13% on first $50
    p50_1000 = 0.05   # 5% on $50.01–$1000
    p1000    = 0.02   # 2% above $1000
    fee      = 0.50   # listing fee

    if sell_price <= 50:
        fee += sell_price * p50
    elif sell_price <= 1000:
        fee += (50 * p50) + \
               (sell_price - 50) * p50_1000
    else:
        fee += (50 * p50) + (950 * p50_1000) + \
               (sell_price - 1000) * p1000
    return fee

print(f"${calc_ebay_fee(9.95):.2f}")   # → $1.79
print(f"${calc_ebay_fee(100.0):.2f}")  # → $9.50
Functions can contain any valid Python: branches, loops, nested calls, list comprehensions — everything.

6.8 — Functions: Common Errors

Common Function Errors

❌ Copy-Paste Error

def celsius_to_fahrenheit(celsius):
    temperature = (9.0 / 5.0) * celsius
    fahrenheit  = temperature + 32
    return fahrenheit      # ✅ correct

def fahrenheit_to_celsius(fahrenheit):
    temperature = fahrenheit - 32
    celsius     = temperature * (5.0 / 9.0)
    return fahrenheit      # ❌ BUG — should be celsius!
No syntax error — Python runs it happily. The function sometimes returns a plausible-looking number, making it very hard to find.
Also watch for the copy-paste sum example: adding num3 * num4 but forgetting to change num3 to num4 in the last term.

❌ Missing Return Statement

def steps_to_feet(num_steps):
    feet_per_step = 3
    feet = num_steps * feet_per_step
    # FORGOT: return feet

feet = steps_to_feet(1000)
print(f"Distance: {feet} feet")  # → Distance: None feet

# Later this might cause:
total = feet + 500  # TypeError: NoneType + int
Python silently returns None. The program doesn't crash here — it crashes later with a confusing error about NoneType.
Golden rules after copy-pasting:
1. Read every line of the pasted code.
2. Double-check every return — which variable are you actually returning?
6.9 — Scope of Variables and Functions

Variable Scope: Local vs. Global

CM_PER_INCH    = 2.54   # ← GLOBAL — visible everywhere
INCHES_PER_FT  = 12    # ← GLOBAL constant

def height_US_to_cm(feet, inches):
    # total_inches is LOCAL — born here, dies when function returns
    total_inches = feet * INCHES_PER_FT + inches
    centimeters  = total_inches * CM_PER_INCH
    return centimeters

feet   = int(input("Feet: "))
inches = int(input("Inches: "))
print(f"Centimeters: {height_US_to_cm(feet, inches)}")

# This would crash with NameError:
# print(total_inches)   ← does not exist outside the function!
# print(centimeters)    ← same — local only

# Function must be DEFINED before it is CALLED:
# get_name()            ← NameError if def is below this line!
# def get_name(): ...
A function's own feet parameter is a different variable from the global feet — same name, different namespace. This can confuse beginners; use distinct names when possible.
Local Variable

Created inside a function. Exists only while the function is running. Invisible outside. Each call gets its own independent copy.

Global Variable

Created outside any function. Visible from its assignment to the end of the file. Accessible inside functions for reading.

Best practice: Use globals only for constants. Avoid modifying global variables inside functions — prefer passing values as parameters and returning results instead.
Parameters are also local variables — created when the function is called, destroyed when it returns.
6.9 — Global Variables

Modifying Global Variables: the global Statement

❌ Without global — creates a local, doesn't work

employee_name = "N/A"

def get_name():
    name = input("Enter name: ")
    employee_name = name   # creates a LOCAL variable!
                           # global is unchanged

get_name()
print(employee_name)    # → N/A  (unchanged)

✅ With global — modifies the global correctly

employee_name = "N/A"

def get_name():
    global employee_name   # ← declare intent to modify global
    name = input("Enter name: ")
    employee_name = name   # now modifies the GLOBAL

get_name()
print(employee_name)    # → Juliet Capulet
Rule: Any assignment inside a function creates a local variable by default — even if a global with the same name exists. Use global x only when you truly must modify a global.
Mutable exception: Modifying list contents — my_list.append(x) — does not require global. Only reassignment (my_list = []) does.
Avoiding global variables makes your functions predictable. A function's behavior should be fully determined by its parameters — no hidden side effects.
6.10 — Multiple Function Outputs

Returning Multiple Values via Tuple

student_scores = [75, 84, 66, 99, 51, 65]

def get_grade_stats(scores):
    """Returns (mean, std_dev) for a list of scores."""
    mean = sum(scores) / len(scores)

    tmp = 0
    for score in scores:
        tmp += (score - mean) ** 2
    std_dev = (tmp / len(scores)) ** 0.5

    return mean, std_dev    # ← returns a TUPLE (no parens needed)
    # equivalent: return (mean, std_dev)
    # also valid:  return [mean, std_dev]  (a list)

# ── Unpacking — multiple assignment at once ───────────────
average, std = get_grade_stats(student_scores)
print(f"Average: {average:.1f}")    # → Average: 73.3
print(f"Std Dev: {std:.1f}")        # → Std Dev: 15.3

# ── Capture as one tuple ──────────────────────────────────
stats = get_grade_stats(student_scores)
print(stats)   # → (73.333..., 15.260...)

# ── get_positions example ────────────────────────────────
def get_positions(number_list):
    first_num  = number_list[0]
    middle_num = number_list[len(number_list) // 2]
    last_num   = number_list[-1]
    return first_num, middle_num, last_num

begin, center, end = get_positions([10, 20, 30, 40, 50])
  • return a, b creates and returns a tuple
  • Parentheses are optional: return (a, b) is identical
  • Could also return a list: return [a, b]
  • Unpacking assigns multiple variables in one statement — must match count
return a, b, [c, d] is valid — returns a 3-element tuple where the third element is a list. Complex containers work too.
What does x, y = get_grade_stats([80, 90, 70]) assign to x?
👆 Tap to reveal
x = 80.0 (the mean). y gets the standard deviation (~8.16). Unpacking assigns left to right from the returned tuple.
6.11 — Recursive Functions

Recursive Functions

def count_down(count):
    if count == 0:          # ← BASE CASE (stops recursion)
        print("Go!")
    else:
        print(count)
        count_down(count - 1)  # ← RECURSIVE CALL

count_down(3)
# 3  → count_down(3) prints 3, calls count_down(2)
# 2  → count_down(2) prints 2, calls count_down(1)
# 1  → count_down(1) prints 1, calls count_down(0)
# Go!→ count_down(0) hits base case, returns

# ── Backwards alphabet ───────────────────────────────────
def backwards_alphabet(curr_letter):
    if curr_letter == "a":     # base case
        print(curr_letter)
    else:
        print(curr_letter)
        prev = chr(ord(curr_letter) - 1)  # 'd' → 'c' → 'b'…
        backwards_alphabet(prev)          # recursive call

backwards_alphabet("d")   # → d  c  b  a
Call count: count_down(5) makes 6 total calls (5 recursive + 1 base case).
Two Requirements for Recursion

1. Base case — the condition that stops the recursion (no self-call).

2. Recursive case — the function calls itself with a simpler version of the problem, moving toward the base case.

  • Each call gets its own local namespace (own copy of parameters)
  • Without a base case → infinite recursion → program crash
  • Most loops can be rewritten as recursion and vice versa
  • Recursion is natural for problems that are self-similar: file trees, fractals, factorial, Fibonacci
Counting down is easier with a loop — but backwards_alphabet shows a case where recursion reads cleanly. Use the right tool for the problem.
6.12 — Documenting Functions

Docstrings & the help() Function

# ── Single-line docstring ─────────────────────────────────
def num_seats(airliner_type):
    """Determines number of seats on a plane."""
    pass   # body here...

# ── Multi-line docstring ──────────────────────────────────
def ticket_price(origin, dest, coach=True, first_class=False):
    """Calculates the price of a ticket between two airports.

    Only one of coach or first_class may be True.
    Arguments:
      origin      -- string: IATA code of origin airport
      dest        -- string: IATA code of destination airport
    Optional keyword arguments:
      coach       -- bool: True for coach pricing (default True)
      first_class -- bool: True for first class pricing
    """
    pass   # body here...

# ── Using help() ──────────────────────────────────────────
help(ticket_price)    # prints docstring nicely formatted
help(str)             # works on built-in types too!
help(range)
help(max)

# ── Your own function ─────────────────────────────────────
def m_to_cm(m):
    """Converts meters to centimeters where 1 m = 100 cm."""
    return 100 * m

help(m_to_cm)   # prints: Converts meters to centimeters...
  • Docstring = string literal as the first statement in the function body
  • Starts and ends with """triple quotes"""
  • Single-line: keep it on one line with the quotes
  • Multi-line: one blank line before closing """; document parameters and return value
help() reads the docstring and prints formatted documentation. Works on any function — user-defined or built-in!
Document every function you write — your future self and teammates will thank you. A good docstring answers three questions: What does it do? What does it take? What does it return?
No blank lines before or after the docstring. The """ must be the literal first statement — not a comment.

6.14 — Namespaces Optional Deep Dive

Namespaces & Scope Resolution

What is a Namespace?

A namespace is a dictionary mapping names → objects. Python maintains separate namespaces for different scopes. Each function call creates its own local namespace that is destroyed on return.

daily_cals = 2300
soda_cals  = 200

def drink_soda(cals_left):
    # Python looks for soda_cals:
    # 1st: local namespace  — not found
    # 2nd: global namespace — found! (200)
    return cals_left - soda_cals

daily_cals = drink_soda(daily_cals)  # → 2100

# Inspect your namespaces:
print(globals())   # all names at global scope
# inside a function:
def show_locals(x):
    y = x * 2
    print(locals())   # {'x': ..., 'y': ...}
LEGB Rule — Scope Resolution Order
  • Local — inside the currently executing function
  • Enclosing — any outer functions (nested function scope)
  • Global — module-level names (outside any function)
  • Built-in — Python's built-in names: int, str, list…
When Python sees a name, it searches L → E → G → B in that order. First match wins. If not found anywhere → NameError.
The same name can exist in multiple namespaces simultaneously. A local tmp inside a function shadows a global tmp — they are independent objects.
If you define a function called list(), it shadows the built-in — your version wins inside that scope. Not a crash, but a very confusing bug.
6.17 — Arbitrary Argument Lists Optional Deep Dive

Flexible Functions: *args and **kwargs

*args — arbitrary positional arguments → tuple

def print_sandwich(bread, meat, *args):
    print(f"{meat} on {bread}", end=" ")
    if len(args) > 0:
        print("with", end=" ")
        for extra in args:
            print(extra, end=" ")
    print()

print_sandwich("sourdough", "turkey", "mayo")
# → turkey on sourdough with mayo

print_sandwich("wheat", "ham", "mustard", "tomato", "lettuce")
# → ham on wheat with mustard tomato lettuce

# *args collects "extra" positional args into a tuple
# bread and meat are required; everything after is optional

**kwargs — arbitrary keyword arguments → dict

def print_sandwich(meat, bread, **kwargs):
    print(f"{meat} on {bread}")
    for category, extra in kwargs.items():
        print(f"  {category}: {extra}")

print_sandwich("turkey", "sourdough", sauce="mayo")
# → turkey on sourdough
# →   sauce: mayo

print_sandwich("ham", "wheat",
               sauce1="mustard", veggie1="tomato")
# → ham on wheat
# →   sauce1: mustard
# →   veggie1: tomato
Ordering rule: Normal params → *args → **kwargs. Both can appear together but must be last in that order.
"args" and "kwargs" are just conventions — any valid identifier works: *toppings, **options. The * and ** are what matter.
