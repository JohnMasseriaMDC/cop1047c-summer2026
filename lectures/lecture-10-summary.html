Chapter 9 · Lists Deep Dive

🧠 Lists Are Really References — Live in Python Tutor

1Tonight's lecture leaned heavily on Python Tutor, a free site that steps through your code line by line and visualizes what's happening in memory. If you missed the live demo, it's worth spending 15 minutes with it on your own — pasting in a script and clicking "Visualize Execution" builds the exact mental model this course keeps coming back to.

The core idea: a variable like myList isn't a box that holds the list — it's a pointer to a list object elsewhere in memory. And the items inside that list aren't the actual values either (in most cases) — they're pointers to other objects. A string like "cat" lives somewhere in memory on its own; the list just holds a reference to it. This is what makes random access to any element fast and consistent-size regardless of what's stored there — the list holds fixed-size pointers, not variable-size values.
list_index_trace.py
animals = ["cat", "dog", "bird", "raptor"]
print(animals[0])   # cat
print(animals[2])   # bird
i = 3
print(animals[i])   # raptor — index 3 is the 4th item
Terminology trap: index 3 is the fourth item in the list. The index is always one less than the position number — index 0 is the first item, index 1 is the second, and so on. Trying to access animals[4] on a 4-item list raises IndexError, because valid indices only run from 0 to len(list) - 1.

Once animals.append("dog") is called, the list grows to 5 items, so animals[4] becomes valid.

🧰 List Methods You'll Actually Use

2A quick-reference set of list operations, worth memorizing as a cheat sheet:

MethodWhat it does
list.append(x)Adds x as a single new item at the end
list.extend(other_list)Adds each item of other_list individually — different from appending the whole list as one nested item
list.insert(i, x)Inserts x before position i
list.remove(x)Removes the first occurrence of value x — raises an error if x isn't found
list.pop()Removes and returns the last item
list.pop(i)Removes and returns the item at index i
list.index(x)Returns the index of the first occurrence of x
list.count(x)Returns how many times x appears
del list[i]Deletes the item at index i (the same del keyword works on variables and dictionary entries too)
Classic mix-up: append vs. extend. my_list.append(other_list) adds other_list as one nested list item. my_list.extend(other_list) adds each of its items individually. If your result has an unexpected list-inside-a-list, this is almost always why.
Real-world pattern: append + pop as a queue. Appending new items to the end and popping from the front (pop(0)) builds a fair first-come-first-served queue — think people lining up at one of several service counters. Checking if my_list (or if not my_list) tells you whether there's anything waiting to be processed.

Use x in list or list.count(x) to safely check whether a value exists before calling list.remove(x) — this avoids the exception remove() raises when the value isn't found.

⚠️ Don't Modify a List While You're Iterating Over It

3This is one of the most common real bugs you'll write, and it's subtle because Python doesn't raise an error — it just silently produces the wrong answer.

broken_negatives.py
# This does NOT change the list — num is just a local copy of each value
for num in nums:
    if num < 0:
        num = 0   # only changes the loop variable, not the list
fixed_negatives.py
# Fixed with enumerate — you get BOTH the index and the value
for i, num in enumerate(nums):
    if num < 0:
        nums[i] = 0   # this actually updates the list

A related, sneakier version of the same trap: removing items from a list while looping over that same list causes items to get skipped, because the list shifts as you remove from it and the loop's internal position doesn't account for the shift.

safe_removal.py
values = [10, 20, 30, 40]
numbers_to_remove = [20, 40]

for val in values.copy():        # iterate over a COPY
    if val in numbers_to_remove:
        values.remove(val)

print(values)   # [10, 30] — correct
The fix, in general: if you need to remove or change items in a list while looping over it, either loop over list.copy() instead of the original, or use enumerate() to update by index rather than removing/reassigning items mid-loop.

🗺️ Nested Lists (2D Lists)

4A list of lists behaves like a table — a tic-tac-toe board, or a spreadsheet with rows and columns.

board.py
board = [
    ["X", "O", "X"],
    ["O", "X", "O"],
    ["X", " ", "O"]
]
print(board[1][1])   # row 1, column 1 -> "X"

for row in board:
    for cell in row:
        print(cell, end=" ")
    print()
len() on a 2D list only gives you the number of rows, not the total number of cells. To count every item, you have to sum the length of each row.

Think of a 2D list the same way you'd think of an Excel spreadsheet: the first index is the row, the second is the column — except Python starts counting both from 0 instead of using letters and 1-based row numbers.

🔀 sort() vs. sorted()

5list.sort() sorts the list in place and returns None — if you try to capture its return value (x = my_list.sort()), x will be None, and this mistake won't be flagged as an error until you try to use x later and it fails.

sort_vs_sorted.py
nums = [30, 10, 20]
nums.sort()              # sorts in place, original order is lost
print(nums)               # [10, 20, 30]

nums2 = [30, 10, 20]
result = sorted(nums2)    # returns a NEW sorted list
print(nums2)               # [30, 10, 20] — original preserved
print(result)              # [10, 20, 30]
Sorting names case-insensitively: since uppercase letters have lower character-code values than lowercase ones (an ASCII/Unicode fact), sorting a plain list of names puts all-uppercase-first names ahead of others regardless of alphabetical order. Passing key=str.lower tells sort()/sorted() to compare using the lowercase version of each name — without actually changing the values stored in the list. Add reverse=True to sort descending instead of ascending.
This is a runtime bug, not a compile-time one. In statically-typed, compiled languages, assigning a function's None-like return to a variable typed as a list would often be caught before the program ever runs. Python is dynamically typed, so this kind of mistake isn't caught until the program actually executes and something downstream fails — one of the real tradeoffs of a flexible, dynamically-typed language.

📖 Dictionaries: Key-Value Lookups

6Unlike lists, dictionaries aren't positional — you access values by key, not by numeric index. Since Python 3.7, dictionaries preserve insertion order (in earlier versions they didn't), so iterating over keys returns them in the order they were added.

OperationWhat it does
key in dictionaryReturns True/False without raising an error — safe way to check existence first
dictionary.get(key, default)Returns the value for key, or default if the key isn't present — no exception, no if needed
dictionary.update(other_dict)Merges another dictionary in — updates existing keys, adds new ones
dictionary.pop(key)Removes key and returns its value (raises an error if the key doesn't exist and no default is given)
dictionary.clear()Empties the dictionary completely, but the dictionary itself still exists
del dictionaryDeletes the entire dictionary object — it no longer exists afterward
.items() / .keys() / .values()Return the key-value pairs (as tuples), just the keys, or just the values
Why get() and in matter for real applications: if you're pulling JSON data back from a web API into a dictionary (weather data, for example), you can't always guarantee every key you expect will be present. Referencing a missing key directly raises an exception that will crash your program if uncaught. dictionary.get("temperature", "not available") handles the missing case in one line, with no try/except or if statement needed.

Dictionaries are the right structure when your data is naturally keyed rather than sequential — pick a list when data is ordered/sequential (spreadsheet rows), a dictionary when you need to look something up by a meaningful key (an airport code, a username, a product SKU).

⚡ List Comprehensions

7List comprehensions aren't required material for this course and won't be tested directly — but they're one of Python's defining features, and worth understanding because you'll see them constantly in real code (and in AI-generated code).

comprehension_basics.py
# The long way
nums = [10, 20, 30]
new_list = []
for i in nums:
    new_list.append(i + 5)
print(new_list)   # [15, 25, 35]

# The same thing, as a list comprehension
new_list = [i + 5 for i in nums]
print(new_list)   # [15, 25, 35]

Read it as: "build a new list from i + 5, for every i in nums." The expression on the left (i + 5) can be anything — including a type conversion:

comprehension_convert.py
nums = [10, 20, 30, 40, 50, 60, 70]
as_strings = [str(n) for n in nums]

typed_in = input("Enter numbers separated by spaces: ")
as_ints = [int(s) for s in typed_in.split()]

Adding an if clause filters which items make it into the new list:

comprehension_filter.py
nums = [3, 5, 23, -1, 456, 1, 6, 83]

evens = [n for n in nums if n % 2 == 0]
negatives = [n for n in nums if n < 0]
doubled_if_big = [n * 2 for n in nums if n > 20]
Where this comes from: the syntax mirrors set-builder notation from mathematics — "the set of all x such that some condition holds." If you've seen that notation before, list comprehensions will feel familiar; if not, that's fine, it's a shorthand for the plain for-loop-and-append pattern shown above.

🌡️ Temperature Conversion: The Lord Kelvin Edition

8Building on the original Celsius/Fahrenheit converter your groups already delivered, this update adds the Kelvin scale — named after Lord Kelvin — and folds in last week's exception-handling material.

Why Kelvin is different: unlike Celsius or Fahrenheit, Kelvin has a true absolute zero — 0 K is the point at which matter has no kinetic energy left, and there is no temperature lower than that. 0 K corresponds to -273.15°C.

Celsius → KelvinK = C + 273.15
Kelvin → CelsiusC = K - 273.15
