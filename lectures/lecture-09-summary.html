🏦 Banker's Rounding — Explained in Full

This is a correction from earlier in the course, finally given the full treatment it deserved: Python's round() and :.2f formatting do not use the "round half up" rule most of us learned in elementary school.

Elementary school rule (what Python does NOT do): if the digit being rounded is 5 or higher, always round up. This rule rounds up more often than it rounds down (5 out of 9 possible nonzero digits vs. 4), which introduces a small systematic bias — historically a real concern for banks calculating interest across millions of accounts.
Banker's rounding (what Python actually does): when a value is exactly halfway, round to the nearest even digit. 0.5 rounds to 0 (even), 1.5 rounds to 2 (even), 2.5 rounds to 2, 3.5 rounds to 4. This evens out the rounding bias over a large number of calculations.
banker_rounding.py
print(round(0.125, 2))   # 0.12  (2 is even, 3 is odd)
print(round(0.375, 2))   # 0.38  (8 is even, 7 is odd)
The floating-point trap: 2.675 looks like it should round to 2.68. It doesn't — it prints 2.67. That's because 2.675 cannot be stored exactly in binary floating point. Internally it's stored as approximately 2.67499999999999982236..., so when Python rounds, it's really rounding 2.6749..., and the digit being rounded is a 4, which rounds down. This is not a Python bug — it's a consequence of how floating-point numbers are represented in binary, and it's worth being aware of any time you're working with money or precise decimals.

This will not be tested with a "what does bankers' rounding output" style question — it's background knowledge, not exam material. But understanding why floating-point math can surprise you is genuinely useful.

Chapter 8 · Exceptions

💥 What Is an Exception?

1An exception is an error detected during the execution of your program. If it's not handled, the program crashes ("terminates") and Python prints a traceback.

Some of the exceptions you've already met without necessarily naming them:

ExceptionTypical Cause
ValueErrorConverting a non-numeric string to int() or float()
ZeroDivisionErrorDividing by zero
IndexErrorAccessing a list or string index that's out of range
KeyErrorAccessing a dictionary key that doesn't exist
EOFErrorReaching end-of-file unexpectedly during input/file reads
TypeErrorUsing an operation on an incompatible type
NameErrorReferencing a variable or function that doesn't exist

🛟 Handling Exceptions with try / except

2Just like if and while, the try statement introduces an indented block. Statements in the try block run one at a time; the moment one raises an exception, control jumps immediately to the matching except clause.

bmi_basic.py
try:
    weight = int(input("Enter weight: "))
    height = int(input("Enter height: "))
    bmi = (weight / (height ** 2)) * 703
    print(f"Your BMI is {bmi:.1f}")
except:
    print("We could not calculate your health information.")

You can target specific exception types instead of catching everything blindly. Multiple exception types can be grouped as a tuple:

bmi_specific.py
try:
    weight = int(input("Enter weight: "))
    height = int(input("Enter height: "))
    bmi = (weight / (height ** 2)) * 703
except (ValueError, TypeError):
    print("Please enter whole numbers only.")
except ZeroDivisionError:
    print("Invalid height entered — must be greater than zero.")
except (NameError, AttributeError):
    print("That function doesn't exist.")
except:
    print("An unexpected error occurred.")
Order matters. Python checks except clauses top to bottom and stops at the first match. Put your most specific clauses first and a bare except: last — if a catch-all comes first, none of the more specific clauses below it will ever run.
C / C++ / Java note: Python's except plays the same role as catch in those languages. If your instructor slips and says "catch" during lecture, that's why — decades of muscle memory from other languages.

🚩 Raising Your Own Exceptions

3You can trigger an existing exception type yourself with raise, and pass it a custom message:

bmi_raise.py
try:
    weight = int(input("Enter weight: "))
    if weight < 0:
        raise ValueError("invalid weight")

    height = int(input("Enter height: "))
    if height <= 0:
        raise ValueError("invalid height")

    bmi = (weight / (height ** 2)) * 703
except ValueError as e:
    print(f"Error: {e}")

Capturing the exception object itself (as e) lets you print exactly what went wrong instead of a generic message — much more useful for debugging, and a good habit if you're logging errors to a file rather than just printing them.

Exceptions cross function boundaries. If a try block calls a function, and that function (or one it calls) raises an exception, the calling try/except still catches it — even several function calls deep. Wrapping logic in functions and calling them from inside a try block is often cleaner than writing all the exception-handling inline, and it's a legitimate way for a function to signal an error back to its caller (an alternative to returning error codes or tuples of (error_code, value)).

🏗️ Building a Custom Exception Class

4You can define your own exception type by inheriting from Exception. This is a preview of class syntax we'll cover formally in two weeks — enough to use it here, not to fully explain the mechanics yet.

grade_checker.py
class InvalidGradeError(Exception):
    def __init__(self, value):
        self.value = value
        super().__init__(f"{value} is not between 0 and 100")

while True:
    entry = input("Enter a grade (or Q to quit): ")
    if entry.upper() == "Q":
        print("Goodbye")
        break
    try:
        grade = int(entry)
        if grade < 0 or grade > 100:
            raise InvalidGradeError(grade)

        if grade >= 90:
            letter = "A"
        elif grade >= 80:
            letter = "B"
        elif grade >= 70:
            letter = "C"
        elif grade >= 60:
            letter = "D"
        else:
            letter = "F"
        print(f"Grade: {letter}")
    except InvalidGradeError as e:
        print(f"Error: {e}")
    except ValueError:
        print("Please enter a whole number.")

When InvalidGradeError(grade) is raised, its __init__ stores the value and calls Exception's own __init__ with a formatted message — that message is what gets printed when you catch it with as e.

Cascading comparisons, largest first: when checking a range with if/elif using >=, start with the largest threshold and work down — otherwise a 95 would match your first ">= 60" branch and never reach the "A" check.

🔒 finally, and a Global Variable Gotcha

5Code inside a finally block always runs — whether or not an exception occurred.

finally_demo.py
def divide(a, b):
    z = -1
    try:
        z = a / b
    except ZeroDivisionError:
        print("Cannot divide by zero.")
    finally:
        print(f"The result is {z}")
Live-caught gotcha (worth knowing): if instead you write z = -1 outside the function without declaring it global, and then try z = a / b inside a try block without a prior local assignment, a division-by-zero on the very first call can leave z completely unassigned inside the function — because the assignment statement's right-hand side is evaluated first, and if that evaluation raises an exception, the assignment never happens. The finally block's attempt to print z then raises its own UnboundLocalError, since Python treats any variable assigned inside a function as local unless you explicitly write global z. The fix: initialize the variable inside the function (or before the try block) so it always has a value, regardless of whether the assignment inside try succeeds.

Chapter 12 · Files

📂 Opening and Reading Files

6open() takes a filename and a mode string, and returns a file object:

ModeMeaning
'r'Read (default if no mode is given)
'w'Write — creates the file, or overwrites it if it exists
'a'Append — writes are added to the end of the existing file
'rb'Read binary — returns a bytes object rather than a string
read_whole_file.py
f = open("notes.txt")     # defaults to read mode
contents = f.read()       # entire file as one big string
f.close()
print(contents)

read() pulls in the entire file as a single string — fine for small files. readlines() instead returns a list of strings, one per line, which is usually more useful for processing a file line by line:

readlines_demo.py
f = open("notes.txt")
lines = f.readlines()
print(lines[0])   # first line
print(lines[1])   # second line
f.close()
Watch for IndexError. If the file has fewer lines than you expect (or is empty), indexing into readlines()'s result past the end raises an IndexError, same as with any list.

✍️ Writing, Appending, and Buffering

7Everything written to a text file is converted to a string first — numbers must be explicitly converted with str() before writing:

write_demo.py
num1, num2 = 5, 7.5
f = open("results.txt", "w")
f.write(str(num1) + " = " + str(num1 + num2) + "\n")
f.close()
Buffering — why your data might not be on disk yet. Python (and the operating system underneath it) buffers writes for efficiency rather than hitting the disk on every single write() call. If you're logging sensor data — say, from a Raspberry Pi Pico — and it's critical that a value survives a power failure, call f.flush() to force Python to hand the data to the OS, and os.fsync(f.fileno()) to force the OS itself to commit it to disk. How often you flush is a tradeoff: flushing every millisecond wears out flash storage; flushing too rarely risks losing data if power is lost.

🗂️ Writing Portable File Paths

8Windows uses backslashes as its directory separator, but backslash is also Python's escape character — so a hardcoded Windows path needs escaping, and won't work on macOS or Linux anyway.

escape_check.py
s = "\t"
print(len(s))   # 1 — the escape sequence counts as ONE character
Exam-relevant: escape sequences like \t, \n, and \\ each count as exactly one character in a string's length, regardless of how many characters you typed to write them.

The portable, OS-independent way to build a path is os.path.join(), which uses the correct separator for whatever operating system the code is actually running on:

portable_path.py
import os
path = os.path.join("logs", "2026", "07", "readings.txt")

Writing machine-independent code — code that runs the same whether it's on Windows, macOS, Linux, or a specific cloud provider's flavor of Linux — is a core software design goal: it keeps you from being locked into one platform.

with_statement.py
with open("data.txt", "r") as f:
    for line in f:
        print(line.strip())
# file is automatically closed here — even if an exception occurred
The with statement is the preferred, modern way to open a file. It guarantees the file is closed when the block exits — no matter whether that's normal completion or an exception — the same guarantee finally gives you, but built in automatically.

📊 Reading CSV Files

9A CSV (comma-separated values) file is a plain text file structured like a spreadsheet — one row per line, fields separated by commas. (If your data itself might contain commas, a tab-separated file, using \t as the delimiter, avoids the conflict.)

read_grades.py
import csv

with open("grades.csv", "r") as csv_file:
    reader = csv.reader(csv_file, delimiter=",")
    for row in reader:
        name = row[0]
        scores = [float(x) for x in row[1:]]
        print(name, sum(scores))

Each row comes back as a list of strings. The line [float(x) for x in row[1:]] is a list comprehension — for every item in the row starting from index 1 (skipping the name), convert it from a string to a float, and build a new list out of the results in one line.

Combining exceptions and files: a robust file-reading program typically wraps the whole read in a try that catches both FileNotFoundError (the file doesn't exist) and ValueError (a value in the file wasn't the type you expected) — giving a clean error message either way instead of a crash.
