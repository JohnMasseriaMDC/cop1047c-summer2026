Chapter 10 · Object-Oriented Programming

🔒 Encapsulation & Abstraction: Why Objects Hide Their Insides

1You've been using objects all along without necessarily calling them that. Every time you call my_string.lower(), my_list.append(), or my_dict.get(), you're calling a method on an object. Python is a purely object-oriented language — strings, lists, dictionaries, floats, everything is an object.

The core idea — a "contract," not a peek under the hood: when you call math.sqrt(41), you don't need to know how it computes a square root (iterative averaging? a hardware floating-point instruction? a lookup table?) — you just need to know it takes a number and returns the square root. That's the contract. This is abstraction: the user of an object is shielded from its implementation details.

Encapsulation is the related idea that an object bundles its data and the operations on that data together, and controls access to that data through its methods rather than exposing it directly.

Design example: storing age vs. storing a birthdate. A person's age changes every year (and depends on today's date), so storing "age" directly as a number is fragile — the value goes stale the moment the calendar turns over. The better design stores the birthdate (which never changes) and computes age on demand through a method. This also matters for a security reason: birthdate is personally identifiable information, something worth being deliberately careful with in any real system that handles guest or customer data.

A "don't store data that doesn't belong to this object" rule applies broadly — a car-on-a-lot object shouldn't store the dealership's total salesperson count, and a restaurant object shouldn't store today's weather. If it's not intrinsic to that one object, it belongs somewhere else in your program.

🏗️ Building a Class: The Time Example

2A class definition includes a constructor (__init__) and any instance methods. self is the placeholder name (by convention — it could technically be anything) for a pointer to the specific object instance a method is operating on. Python passes it automatically when you call a method; you only write it explicitly in the method's own definition.

time_class.py
class Time:
    def __init__(self):
        self.hours = 0
        self.minutes = 0

    def print_time(self):
        print(f"{self.hours:02d}:{self.minutes:02d}")

time1 = Time()
time1.hours = 5
time1.minutes = 30
time1.print_time()   # 05:30
If you forget self in a method definition, you'll get an error like takes 0 positional arguments but 1 was given — because Python is always silently passing the object reference as the first argument, whether or not you declared a parameter to receive it.

Other languages use different names for this same concept — Java and C++ typically use this. Python's convention is self. Python's equivalent of Java's toString() method (for defining how an object prints/displays itself) is __str__.

⚠️ The Aliasing Gotcha

3This is one of the most important — and most commonly misunderstood — behaviors in Python. Assigning one variable to another doesn't copy the object; it creates a second name pointing to the same object.

aliasing.py
time1 = Time()
time1.hours = 7

time2 = time1     # NOT a new Time object — an alias
print(time2.hours)   # 7, because time2 IS time1
Real-world version: a cabin check-in system. If cabin3 = cabin1, then cabin3 isn't a new cabin — it's another name for the exact same cabin object. Checking a guest into "cabin3" actually updates cabin1's occupancy, because there was never a second cabin created. If you need an independent copy rather than an alias, you have to explicitly construct a new object rather than just assigning one variable to another.

🌐 Class Attributes vs. Instance Attributes

4An attribute defined inside __init__ with self. is an instance attribute — every object gets its own separate copy. An attribute defined at the class level (outside any method, no self) is a class attribute — it's shared by every instance of that class.

marathon_runner.py
class MarathonRunner:
    race_distance_km = 42.195   # class attribute — shared by ALL runners

    def __init__(self, speed):
        self.speed = speed      # instance attribute — unique per runner

runner1 = MarathonRunner(7.5)
runner2 = MarathonRunner(8.0)
print(runner1.race_distance_km)   # 42.195
print(runner2.race_distance_km)   # 42.195 — same value, shared
Fallback lookup behavior: if you reference self.some_name inside a method and no instance attribute by that name exists, Python falls back to checking the class for a class attribute with that name. This is genuinely useful (a GMT-offset example used this to apply one shared offset across every Time instance), but it can also mask bugs — be intentional about which kind of attribute you actually mean to use.

To update a class attribute, reference it through the class name (MarathonRunner.race_distance_km = ...), not through an instance — updating via the class name changes the value for every existing instance at once.

🎛️ Default and Named Parameters

5Constructor parameters can have default values, and can be passed by name in any order — which is exactly why descriptive parameter names (not single letters like a, b, c) matter.

employee.py
class Employee:
    def __init__(self, name, wage=8.25, hours=20):
        self.name = name
        self.wage = wage
        self.hours = hours

todd = Employee("Todd")                          # uses both defaults
trisha = Employee("Trisha", wage=12.50, hours=40) # overrides both
john = Employee(hours=40, wage=9, name="John")    # named args, any order

name has no default, so it's a required (positional) parameter — every Employee must be given one. wage and hours can be skipped, overridden positionally, or overridden by name in any order, as the john example shows.

🧬 Inheritance: Building on a Base Class

6A class can inherit (or "derive") from another class, gaining everything the parent (base class) defines, and adding its own attributes and methods on top.

grocery_inheritance.py
class Item:
    def __init__(self):
        self.name = ""
        self.quantity = 0

    def set_name(self, name):
        self.name = name

    def set_quantity(self, quantity):
        self.quantity = quantity

    def display(self):
        print(f"{self.name}: {self.quantity}")

class Produce(Item):
    def __init__(self):
        Item.__init__(self)      # calls the parent's constructor
        self.expiration = ""

    def set_expiration(self, date):
        self.expiration = date

apples = Produce()
apples.set_name("Apples")
apples.set_quantity(40)
apples.set_expiration("2027-01-01")
apples.display()   # "Apples: 40" — display() only knows about Item's fields
Terminology, all describing the same relationship: Produce "derives from," "inherits from," or "extends" Item. Item is the base class. Any class can itself be a base class for something else — a grocery store's hierarchy might go Item → Produce → Fruit, with each level in the chain needing to call its immediate parent's __init__, cascading all the way up. This kind of class hierarchy is often documented with UML (Unified Modeling Language) diagrams — a notation used for both object designs and database schemas.
Every class already inherits from something. Even a class with nothing in parentheses after its name (class Item:) implicitly inherits from Python's built-in object class — there's no such thing as a completely standalone class in Python.
Tooling

🔗 Sharing Code with GitHub Gist

7A GitHub Gist is a lightweight way to share a code snippet via a URL — useful for breakout collaboration without needing a full repository. Creating one requires a free GitHub account.

Secret gists are not private. A "secret" gist just isn't publicly listed or searchable — anyone with the URL can still view it (though they can't edit it without their own account). Never put real secrets (passwords, API keys, personal data) into a gist, secret or not. GitHub also keeps a full revision history: editing a secret out of a gist doesn't erase it — the original version is still reachable through the revision history. Treat anything ever committed as permanently visible.
Chapter 11 · Modules

📦 Modules: Splitting Code Across Files

8You've been using modules since early in the course — math, sys, os are all modules. In its simplest form, a module is just another .py file that you import.

my_funcs.py
def factorial(n):
    x = 1
    for i in range(1, n + 1):
        x *= i
    return x
main.py
import my_funcs
print(my_funcs.factorial(7))   # 5040 — must prefix with the module name

# OR:
from my_funcs import factorial
print(factorial(7))   # 5040 — no prefix needed
Module names are case-sensitive, and Python looks for the file by its exact name plus .py — import my_funcs looks for my_funcs.py in the same directory. For consistency, it's simplest to keep module filenames all lowercase.
Avoid from module import *. It's valid syntax, but importing everything from a module risks name collisions — if that module defines a function with the same name as one of yours (or as one from another module you've imported), one silently overwrites the other. Import only the specific names you need, or use the module-prefixed form.

🗂️ Packages: Directories of Modules

9A package is a directory containing modules (and possibly sub-packages). Historically an __init__.py file was required to mark a directory as a package; this is no longer strictly enforced in current Python, but any code inside __init__.py still runs automatically whenever the package is imported, which makes it a useful place for package-level setup.

package_imports.py
import ascii_art.canvas
ascii_art.canvas.draw_canvas()

from ascii_art.figures import cow
cow.draw()

from ascii_art.figures.cow import draw
draw()   # no prefix needed at all

🔐 A Quick Look at Hashing and HTTPS

This section is background context on the hashlib module, not required material for the final. A hash function takes in data and produces a fixed-size number (a "hash") that changes dramatically if even a single bit of the input changes.

hashlib_example.py
import hashlib

text = "to be or not to be"
digest = hashlib.md5(text.encode("utf-8")).hexdigest()
print(digest)
Why this matters in the real world: hashing algorithms underpin how HTTPS certificates are trusted. A certificate authority hashes a website's certificate and encrypts that hash with its private key. Your browser, which ships with a built-in list of trusted certificate authorities' public keys, can decrypt that hash and independently compute its own hash of the certificate — if the two match, the certificate hasn't been tampered with. MD5 itself is no longer considered secure for this purpose; modern systems use algorithms like SHA-256.

🚪 The if __name__ == "__main__": Guard

10Every Python file has a built-in variable __name__. When you run a file directly, __name__ is set to "__main__". When that same file is instead imported as a module by another program, __name__ is set to the module's filename instead.

guarded_module.py
def factorial(n):
    x = 1
    for i in range(1, n + 1):
        x *= i
    return x

if __name__ == "__main__":
    n = int(input("Enter a number: "))
    print(factorial(n))
Without this guard, importing a module can trigger code you didn't intend to run. If the input/print block above weren't wrapped in the if __name__ == "__main__": check, simply importing this file into another program would immediately prompt for input and print a result — before your own program even got a chance to run. This is exactly the pattern your DNA Sequence Search program should use: define your functions and classes first, then guard your "run the program" code at the bottom with this check.

🧪 Virtual Environments

11Not every module is built into Python — third-party packages (like requests) need to be installed with pip. Installing packages directly into your system Python can require elevated privileges and risks version conflicts between projects. A virtual environment gives each project its own isolated set of installed packages.

venv_commands.sh
# Create a virtual environment (commonly named .venv)
python3 -m venv .venv

# Activate it (macOS/Linux)
source .venv/bin/activate

# Now pip installs go into this isolated environment
pip install requests

# Leave the virtual environment
deactivate
Naming convention: a virtual environment folder starting with a dot (like .venv) is hidden from a plain ls on macOS/Linux — don't assume it's missing just because you don't see it listed. Visual Studio Code automatically detects a virtual environment in your project folder and can use it to manage which packages are available to your code.

You can browse the universe of installable third-party packages at PyPI (the Python Package Index) — there are hundreds of thousands of published projects there.
