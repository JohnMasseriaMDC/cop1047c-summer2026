📝 Midterm Review — What Tripped People Up

A few of the most-missed questions, worth re-tracing:

R1 The ternary operator, combined with +=

Given: if age < 18: x = x + 5 else: x = x + 1 — which one-liner is equivalent?

ternary_review.py
x = x + 5 if age < 18 else x + 1     # correct

x += 5 if age < 18 else x + 1        # WRONG — this is a
                                      # regular if/else pattern,
                                      # not a true ternary
Why it's tricky: a true ternary always has the shape value_if_true if condition else value_if_false — one expression, evaluated once. Combining it with += breaks that shape. Know the ternary by its structure, not just by the keywords if/else appearing on one line.

R2 range() stop value, again

To get every number from 11 to 21 (inclusive), stepping by 2, you need range(11, 22, 2) — the stop must be past 21, since 21 itself would never be produced by a range that stops at 21. This was the most-missed question on the exam. If a range question shows up on the final, trace it on paper before choosing.

R3 Command-line arguments (sys.argv)

Given python3 analyze.py results.csv, sys.argv is a list where argv[0] is the script name itself ("analyze.py") and argv[1] is the first real argument ("results.csv"). This wasn't heavily emphasized in earlier lectures — if it's unfamiliar, it's worth a quick look before the final, since a similar question may return.

Topic 1

🧵 Strings Are Immutable Sequences

  • A string is an ordered sequence of characters — indexed from 0 to len(s) - 1, or -1 (last) down to -len(s) (first).
  • Strings cannot be changed in place. Any "modification" actually builds a brand-new string in a new memory location — you can prove this with id() before and after.
  • Referencing an index beyond the string's length raises an IndexError, whether you go too far positive or too far negative.
immutable.py
word = "Python"
print(id(word))          # some memory address

word = "J" + word[1:]    # builds a NEW string: "Jython"
print(word)               # Jython
print(id(word))          # a DIFFERENT address —
                           # the old "Python" string still exists
                           # until Python's garbage collector reclaims it
Topic 2

✂️ Slicing: s[start:stop:step]

Slicing pulls out part of a string. Like range(), the stop index is never included.

SliceMeaning
s[7:23]Characters 7 through 22
s[1:]From index 1 to the end
s[:-1]Everything except the last character
s[:]A full copy of the string
s[::2]Every other character (step = 2)
s[::-1]Reverses the string — a handy trick
Slicing is forgiving on the high end: asking for a stop index past the actual end of the string is not an error — Python just stops at the end. (Asking for a plain out-of-range index like s[100] is still an error; only slice ranges get this leniency.)
Real-world example from class: parsing a URL like https://en.wikipedia.org/wiki/Python into its protocol, domain, and path. Hard-coded slice positions work but only for that exact URL. A more robust approach uses find() to locate the :// and the surrounding slashes dynamically, so the same code works for any well-formed URL — including checking that each find() actually succeeded (a result of -1 means "not found," so you should treat that as a malformed URL and handle it instead of proceeding).
Topic 3

📐 f-string Field Width, Alignment & Precision

formatting.py
name = "Bob"
score = 42

print(f"{name:<10}|")     # left-justified,  width 10 (default for strings)
print(f"{name:>10}|")     # right-justified, width 10
print(f"{name:^10}|")     # centered,        width 10
print(f"{score:>6}")      # right-justified (default for numbers), width 6
print(f"{score:06}")      # zero-padded:  000042
print(f"{3.14159:8.2f}")  # width 8, 2 decimal places:  "    3.14"
SymbolAlignmentDefault for
<Left-justifiedStrings
>Right-justifiedNumbers
^Centered—
Width is a minimum, not a maximum. If the value is longer than the specified width, Python does not truncate it — the width is simply exceeded. If you need a hard maximum, slice the string yourself before formatting it.
Why this matters: aligned, padded output is how receipts, reports, and ledgers line up their columns — and it's the same logic used when generating a formatted email body, an HTML table, or any fixed-width text report. A custom fill character can replace the default space (e.g. dashes or = signs as a section divider, or asterisks on a printed check to prevent tampering).
Correction

🔢 How Python Actually Rounds

An earlier lecture described Python's rounding the way most of us learned it in elementary school: 5 through 9 rounds up, 0 through 4 rounds down. That description was correct for what most of us were taught, but it is not what Python actually does.

  • Python uses "banker's rounding" (round-half-to-even): when a value is exactly halfway, Python rounds to whichever neighbor is even, not always up.
  • Why: the elementary-school rule rounds up 5 times out of 9 possible digits (5,6,7,8,9) and down only 4 times out of 9 (1,2,3,4) — 0 needs no rounding. Applied over many calculations (like years of interest payments), that asymmetry consistently overpays by fractions of a cent. Rounding to the nearest even digit balances the up-rounds and down-rounds evenly over time.
A full worked demonstration of banker's rounding — including how floating-point storage can make a rounding result look "wrong" even when the rule is applied correctly — is coming in an upcoming lecture. For now, know that the concept exists and why it exists; the mechanics will follow.
Topic 4

🧰 The Core String Toolbox

4a Searching: find, rfind, count, in

CallReturns
s.find("x")Index of the first occurrence, or -1 if not found
s.rfind("x")Index of the last occurrence (searches from the end)
s.count("x")How many times "x" appears
"x" in sTrue/False — is it a substring?
Always check for -1. find() returning -1 means "not found" — but -1 is also a valid negative index, so blindly using an unchecked find() result to slice a string won't crash, it'll just silently give you the wrong slice. Test if position >= 0: before trusting the result.

4b Transforming: replace, strip/lstrip/rstrip, upper/lower/title/capitalize

chaining.py
raw = "  Bob Smith  "
clean = raw.strip().title()      # "Bob Smith" — chained calls
print(clean)

digits = "(555) 123-4567".replace("(", "").replace(")", "") \
                          .replace(" ", "").replace("-", "")
print(digits)                    # "5551234567"
Chaining works because each method returns a brand-new string (never modifies the original) — so you can immediately call another method on that result. replace() defaults to replacing every match; pass a count as the third argument to limit it. None of these methods change the original string — if you don't assign the result to a variable, it's simply discarded.

4c Testing: isdigit, isalpha, isalnum, isspace, startswith, endswith

These return True/False and are useful for validating input before converting it. isdigit() requires every character to be a digit — a decimal point makes it False, so it can't validate a float string on its own. Case-insensitive comparisons (e.g. accepting "quit", "QUIT", or "Quit") are usually done by calling .lower() on the input first, then comparing to a lowercase literal.

4d Splitting and joining

split_join.py
name = "Martin Luther King Jr"
parts = name.split()                  # default delimiter: whitespace
print(parts)                          # ['Martin', 'Luther', 'King', 'Jr']

row = "apple,12,1.99"
fields = row.split(",")               # CSV parsing
print(fields)                         # ['apple', '12', '1.99']

rejoined = ", ".join(parts)           # the reverse of split
print(rejoined)                       # "Martin, Luther, King, Jr"
Unusual syntax to remember: the delimiter calls .join(), not the list — it's "delimiter".join(list), backwards from how most other languages do it. Every item in the list must already be a string, or join() raises a TypeError.
Topic 5

⚖️ Comparing Strings

  • == checks that every character matches, in order. Different lengths can never be equal.
  • < / > compare character by character (alphabetical / dictionary order). Uppercase letters sort before lowercase letters ("A" < "a"), and a shorter string sorts before a longer string that starts the same way ("CAT" < "CATEGORY").
  • Comparisons are case-sensitive — "Quit" != "quit". To accept either, normalize the input first: user_input.lower() == "quit".
Real-World Application

🌐 JSON: How Programs Exchange Data

JSON (JavaScript Object Notation) is the standard format websites and APIs use to send structured data. It looks almost identical to a Python dictionary, and Python's built-in json module converts between the two.

json_demo.py
import json

text = '{"name": "Alice", "age": 30, "isActive": true}'
data = json.loads(text)        # parses the JSON string into a dict

print(type(data))              # <class 'dict'>
print(data["name"])            # Alice
print(type(data["isActive"]))  # <class 'bool'> — JSON's true/false
                                # become real Python booleans
Live demo: we made a real HTTP request (using the requests library) to a public UK carbon-intensity API, got back JSON data, converted it with json.loads(), and pulled out specific fields — region, intensity — exactly the way you'd read any API response. This is genuinely how most modern applications talk to each other: request data over the web, get JSON back, parse it into a dictionary, and use the values. Third-party libraries like requests (for web calls) usually require a virtual environment and pip install, since they don't ship with Python by default — we'll cover environment setup in more detail in a future lecture.
