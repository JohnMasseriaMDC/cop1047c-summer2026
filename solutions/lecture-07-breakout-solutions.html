Breakout Activity #1

🛠️ Miles ↔ Kilometers Converter

🛠️
Breakout #1  ·  8 minutes
Miles ↔ Kilometers Converter
def / return parameters f-strings
Problem Recap
  • Define miles_to_km(miles) — convert miles to km using 1 mile = 1.60934 km
  • Call it for 5, 26.2, and 100 miles; print results to 2 decimal places
  • Bonus A: Write km_to_miles(km) and verify 42.195 km = 26.2 miles
  • Bonus B: Call both in a nested expression and explain the result
1Base Solution
breakout1_solution.py 
KM_PER_MILE = 1.60934    # constant defined at global scope — UPPER_CASE convention

def miles_to_km(miles):
    """Converts a distance in miles to kilometers.

    Arguments:
      miles -- numeric distance in miles (int or float)
    Returns:
      Equivalent distance in kilometers (float)
    """
    return miles * KM_PER_MILE    # one clean computation, returned directly

# ── test calls ────────────────────────────────────────────────────────────────
distances = [5, 26.2, 100]

for d in distances:
    print(f"{d} miles = {miles_to_km(d):.2f} km")
Output
5 miles = 8.05 km
26.2 miles = 42.16 km
100 miles = 160.93 km
2Line-by-Line Explanation
  • Line 1 KM_PER_MILE = 1.60934 — We define the conversion factor as a global constant using UPPER_CASE. This makes the magic number visible and reusable. If the precision ever needs updating, there's exactly one place to change it.
  • Line 3–10 Docstring — Describes what the function does, what it takes, and what it returns. This is what help(miles_to_km) will display.
  • Line 11 return miles * KM_PER_MILE — We don't need a temporary variable here. The expression is simple enough to return directly. The function accesses KM_PER_MILE from the global namespace (it's not defined locally).
  • Lines 14–17 A list + for loop — Rather than three separate print statements, we store the test distances in a list and loop. This is cleaner and easier to extend. The :.2f format spec rounds the output to 2 decimal places.
3Common Mistakes
❌ Forgot the return statement
def miles_to_km(miles):
    result = miles * 1.60934
    # missing: return result!

x = miles_to_km(5)
print(x)   # → None
✅ Always return the value
def miles_to_km(miles):
    result = miles * 1.60934
    return result    # ← required!

x = miles_to_km(5)
print(x)   # → 8.0467
❌ Magic number buried in code
def miles_to_km(miles):
    return miles * 1.60934
# What is 1.60934? Hard to know
# without context. Hard to update.
✅ Named constant at top of file
KM_PER_MILE = 1.60934

def miles_to_km(miles):
    return miles * KM_PER_MILE
# Self-documenting; one place to update
⭐ Bonus A — km_to_miles() + Inverse Verification
breakout1_bonus_a.py 
KM_PER_MILE  = 1.60934
MILES_PER_KM = 1.0 / KM_PER_MILE   # derive from the same source — no separate magic number

def miles_to_km(miles):
    """Converts miles to kilometers."""
    return miles * KM_PER_MILE

def km_to_miles(km):
    """Converts kilometers to miles."""
    return km * MILES_PER_KM

# Verify: a marathon is 42.195 km — should be approximately 26.2 miles
marathon_km = 42.195
print(f"42.195 km = {km_to_miles(marathon_km):.1f} miles")   # → 42.195 km = 26.2 miles
Output
42.195 km = 26.2 miles
⭐ Bonus B — Nested Call & the Identity Round-Trip
breakout1_bonus_b.py 
# Passing the result of km_to_miles() directly into miles_to_km()
# This is a "round-trip" — convert to km, then back to miles
result = miles_to_km(km_to_miles(100))
print(f"{result:.4f}")    # → 100.0000

# Python evaluates inside-out:
#   Step 1: km_to_miles(100)  → 62.1371 miles
#   Step 2: miles_to_km(62.1371) → 100.0 km
#
# This demonstrates that the two functions are INVERSE OPERATIONS.
# In math:  f(g(x)) = x  when f and g are inverses of each other.
Output
100.0000
What this teaches: Hierarchical function calls evaluate inside-out — exactly like int(input()) that you've been using all semester. When two functions are inverses of each other, composing them (calling one inside the other) returns the original value.
Key Takeaway — Breakout #1

A well-written function has one job, a clear name, a descriptive docstring, and a single return statement that sends back exactly what the caller needs. Constants belong at the top of the file — not buried inside a function body. The caller decides what to do with the returned value.

Breakout Activity #2

🏎️ Speed Zone Classifier

🏎️
Breakout #2  ·  10 minutes
Speed Zone Classifier
branches multiple returns while loop
Problem Recap
  • Write classify_speed(mph) that returns a string describing the speed zone
  • Less than 25 mph → "School Zone / Slow"
  • 25 to 64 mph → "Normal / Highway"
  • 65 or above → "Fast / Interstate"
  • Test with 15, 45, and 90 mph
  • Bonus: Read from input() in a loop until -1
1Base Solution
breakout2_solution.py 
def classify_speed(mph):
    """Returns a speed zone classification for the given speed in mph.

    Arguments:
      mph -- numeric speed in miles per hour (int or float)
    Returns:
      A string describing the speed zone:
        'School Zone / Slow'   if mph < 25
        'Normal / Highway'     if 25 <= mph < 65
        'Fast / Interstate'    if mph >= 65
    """
    if mph < 25:
        return "School Zone / Slow"
    elif mph < 65:             # implied: mph >= 25 (already ruled out)
        return "Normal / Highway"
    else:                      # implied: mph >= 65
        return "Fast / Interstate"

# ── test calls ────────────────────────────────────────────────────────────────
test_speeds = [15, 45, 90]

for speed in test_speeds:
    zone = classify_speed(speed)
    print(f"{speed} mph → {zone}")
Output
15 mph → School Zone / Slow
45 mph → Normal / Highway
90 mph → Fast / Interstate
2Design Decisions Explained
  • if mph < 25 Check the lower boundary first. The conditions must be checked in order. Once we know mph is NOT less than 25, the elif inherits that knowledge — we only need to check the upper boundary of the next range.
  • elif mph < 65 No need to write elif mph >= 25 and mph < 65 — the elif only runs if the previous condition was False, which already guarantees mph >= 25. Writing the redundant condition is legal but unnecessary.
  • else The else catches everything remaining — by elimination, we know mph must be >= 65. No explicit condition needed. Always use else as the final fallback so no input goes unclassified.
  • 3 returns One return per branch is valid and clear. Python exits the function at the first return it hits. This function will always return exactly one of the three strings — never None, never nothing.
3Common Mistakes — Condition Order Matters
What happens if you flip the order of conditions? This is one of the most common bugs in functions with branches. Let's trace through it.
❌ Wrong condition order
def classify_speed(mph):
    if mph >= 65:          # checked first
        return "Fast"
    elif mph >= 25:        # checked second
        return "Normal"
    else:
        return "Slow"
# This version actually works correctly!
# But read below for the truly dangerous flip...
❌ Dangerous: boundary in wrong direction
def classify_speed(mph):
    if mph < 65:           # ← catches 15, 45, AND 90!
        return "Slow"      # 90 mph → "Slow" (WRONG!)
    elif mph < 25:         # unreachable — never runs
        return "Normal"
    else:
        return "Fast"
Trace 90 mph through the broken version:
if 90 < 65 → False  |  elif 90 < 25 → False  |  else → "Fast" — actually this one works by accident. Now trace 45 mph: if 45 < 65 → True → returns "Slow". Wrong! A 45 mph highway driver is not "Slow."
⭐ Bonus — Input Loop Until -1 (Sentinel Pattern)
breakout2_bonus.py 
def classify_speed(mph):
    """Returns a speed zone classification for the given speed in mph."""
    if mph < 25:
        return "School Zone / Slow"
    elif mph < 65:
        return "Normal / Highway"
    else:
        return "Fast / Interstate"

# ── Sentinel loop (-1 to quit) ────────────────────────────────────────────────
print("Speed Zone Classifier — enter -1 to quit\n")

speed = float(input("Enter speed (mph): "))     # priming read

while speed != -1:
    zone = classify_speed(speed)
    print(f"  {speed:.0f} mph → {zone}\n")
    speed = float(input("Enter speed (mph): "))  # update at end of loop

print("Goodbye!")
Sample Session
Speed Zone Classifier — enter -1 to quit

Enter speed (mph): 15
  15 mph → School Zone / Slow

Enter speed (mph): 55
  55 mph → Normal / Highway

Enter speed (mph): -1
Goodbye!
Sentinel pattern: We read the first value before the loop (the "priming read"), then read again at the end of the loop body. This ensures we check the sentinel value (-1) at the top of the while condition before processing it. Notice how classify_speed() itself is unchanged — the loop logic lives outside the function, keeping each piece of code focused on one job.
Key Takeaway — Breakout #2

A function with branches must have a return statement in every possible path — otherwise some inputs would cause the function to return None silently. The order of if / elif / else conditions matters: once a condition is True, Python skips the rest. Carefully trace through edge cases (exactly 25 mph, exactly 65 mph) to confirm boundary values land in the right zone.

Breakout Activity #3

📊 List Statistics — Min, Max & Range

📊
Breakout #3  ·  12 minutes
List Statistics — Min, Max & Range
multiple returns tuple unpacking docstrings
Problem Recap
  • Write min_max_range(numbers) returning three values: minimum, maximum, and range (max − min)
  • Include a complete docstring
  • Use unpacking to receive all three values into separate variables
  • Test with [34, 12, 78, 5, 99, 23]
  • Bonus A: Call help(min_max_range) to see the docstring displayed
  • Bonus B: Rewrite without Python's built-in min() and max()
1Base Solution — Using Built-in min() / max()
breakout3_solution.py 
def min_max_range(numbers):
    """Returns the minimum, maximum, and range of a list of numbers.

    Arguments:
      numbers -- a non-empty list of numeric values (int or float)
    Returns:
      A tuple (minimum, maximum, range) where range = maximum - minimum
    """
    minimum = min(numbers)          # Python's built-in min() scans the list
    maximum = max(numbers)          # Python's built-in max() scans the list
    return minimum, maximum, maximum - minimum   # tuple — no parentheses needed

# ── test ──────────────────────────────────────────────────────────────────────
scores = [34, 12, 78, 5, 99, 23]

# Unpacking — three variables receive the three returned values, left to right
mn, mx, rng = min_max_range(scores)

print(f"Min:   {mn}")
print(f"Max:   {mx}")
print(f"Range: {rng}")

# ── Bonus A: call help() to display the docstring ─────────────────────────────
# help(min_max_range)
Output
Min:   5
Max:   99
Range: 94
2Why This Works — Tuples and Unpacking
  • return a, b, c The comma creates a tuple. return minimum, maximum, maximum - minimum is identical to return (minimum, maximum, maximum - minimum). Python sees the comma and packages the values into a tuple automatically.
  • mn, mx, rng = ... Unpacking assigns left-to-right. The first element of the returned tuple goes to mn, the second to mx, and the third to rng. The number of variables on the left must exactly match the number of values in the tuple.
  • Docstring Multi-line docstring format: First line is a one-sentence summary. Then a blank line, then structured sections documenting Arguments and Returns. This is the format that help() displays beautifully.
  • min() / max() Python's built-ins are your friends. min(numbers) and max(numbers) accept any iterable. The Bonus B solution shows how to replicate this yourself — which is a great way to understand what the built-ins are doing under the hood.
3Bonus A — What help() Displays
help(min_max_range) output

⭐ Bonus B — Manual Loop Version (No Built-ins)

This version finds the minimum and maximum manually, using the same technique that min() and max() use internally: start by assuming the first element is both the min and max, then scan the rest of the list to update.

breakout3_bonus_b.py 
def min_max_range(numbers):
    """Returns the minimum, maximum, and range of a list of numbers.
    Uses a manual loop — no built-in min() or max() used.

    Arguments:
      numbers -- a non-empty list of numeric values (int or float)
    Returns:
      A tuple (minimum, maximum, range) where range = maximum - minimum
    """
    # Start by assuming the first element is both the smallest and the largest.
    # This is the standard technique for finding min/max in a single pass.
    minimum = numbers[0]
    maximum = numbers[0]

    for num in numbers:        # iterate over every element
        if num < minimum:
            minimum = num      # found a new minimum — update
        if num > maximum:
            maximum = num      # found a new maximum — update
        # note: using two separate ifs (not if/elif) so a value can
        # update both minimum and maximum independently (though in
        # practice it can only be one or the other, not both)

    return minimum, maximum, maximum - minimum    # same return as the built-in version

# ── test — output is identical ────────────────────────────────────────────────
scores = [34, 12, 78, 5, 99, 23]
mn, mx, rng = min_max_range(scores)

print(f"Min:   {mn}")    # → 5
print(f"Max:   {mx}")    # → 99
print(f"Range: {rng}")   # → 94

# ── trace through the first few iterations ────────────────────────────────────
# numbers[0] = 34   → minimum = 34, maximum = 34
# num = 34  → 34 < 34? No. 34 > 34? No.    (starting element, no change)
# num = 12  → 12 < 34? Yes → minimum = 12. 12 > 34? No.
# num = 78  → 78 < 12? No. 78 > 34? Yes → maximum = 78.
# num = 5   → 5 < 12? Yes → minimum = 5.  5 > 78? No.
# num = 99  → 99 < 5? No. 99 > 78? Yes → maximum = 99.
# num = 23  → 23 < 5? No. 23 > 99? No.   (no change)
# Result: minimum = 5, maximum = 99, range = 94  ✅
Output
Min:   5
Max:   99
Range: 94
Why start with numbers[0] instead of 0? If we initialized minimum = 0 and all values in the list were positive, the final minimum would incorrectly be 0. Starting with an actual element from the list guarantees we never report a minimum or maximum that isn't in the data.
4Thinking About Edge Cases
edge_cases.py 
mn, mx, rng = min_max_range([7])          # single element
print(mn, mx, rng)                         # → 7  7  0   (range is 0 — makes sense)

mn, mx, rng = min_max_range([-10, -3, -7]) # all negatives
print(mn, mx, rng)                         # → -10  -3  7

mn, mx, rng = min_max_range([42, 42, 42])  # all identical
print(mn, mx, rng)                         # → 42  42  0

# min_max_range([])   ← empty list would crash with IndexError
#                       in the manual version, and ValueError in
#                       the built-in version. For production code
#                       you'd guard against this — topic for later!
Empty list guard: Neither version handles an empty list gracefully. For now this is fine — but when you learn about exceptions (Chapter 8), you'll know how to raise a meaningful error if someone calls min_max_range([]).
5Alternate: Capture as One Tuple
capture_as_tuple.py 
# You don't have to unpack immediately — capture as one tuple variable
stats = min_max_range([34, 12, 78, 5, 99, 23])
print(stats)           # → (5, 99, 94)
print(stats[0])        # → 5   (index into the tuple)
print(stats[1])        # → 99
print(stats[2])        # → 94

# Both are valid:
mn, mx, rng = min_max_range(scores)  # unpack right away
stats = min_max_range(scores)        # or keep as a tuple
Key Takeaway — Breakout #3

A function can return multiple values by packing them into a tuple — the comma is all that's needed, parentheses are optional. Unpacking lets you assign all returned values in a single, readable statement. Always write a docstring that answers: what do I take? what do I return? Bonus B demonstrated that Python's built-in functions use the same basic algorithms you can write yourself — understanding the loop version helps you trust and use the built-in version confidently.
