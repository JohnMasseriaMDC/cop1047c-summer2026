Breakout 1 · ~6:25 PM · 10 minutes

🛳️ The Cabin Class

Problem recap: Define Cabin with __init__ creating number/guest/deck defaults, add print_status(), create two instances, then predict the alias behavior of c3 = c1. Stretch: a check_in() method that refuses an occupied cabin.

Base solution — annotated

cabin_solution.py
class Cabin:                                   # Initial-caps class name (convention)
    def __init__(self):                        # constructor runs on every Cabin()
        self.number = ""                       # self = the instance being built
        self.guest = "Vacant"                  # sensible default state
        self.deck = 0

    def print_status(self):                    # instance method: self is required
        print(f"Cabin {self.number} (deck {self.deck}): {self.guest}")

    def check_in(self, name):                  # STRETCH
        if self.guest == "Vacant":
            self.guest = name                  # mutate this instance's state
        else:
            print(f"Cabin {self.number} is occupied by {self.guest}!")

c1 = Cabin()                                   # instantiation #1
c1.number = "A101"
c1.deck = 7
c1.guest = "Rivera"

c2 = Cabin()                                   # instantiation #2 - separate object
c2.number = "B202"
c2.deck = 8                                    # guest stays "Vacant" from __init__

c1.print_status()
c2.print_status()

c3 = c1                                        # TASK 4: alias, NOT a new instance
c3.guest = "Dubois"
c1.print_status()                              # c1 changed too - one object, two names
Cabin A101 (deck 7): Rivera Cabin B202 (deck 8): Vacant Cabin A101 (deck 7): Dubois

Task 4 — expected answer

c1.print_status() now shows Dubois. c3 = c1 copies the reference, not the object — identical to the Chapter 9 list-alias trap. This is the single most important 60 seconds of the debrief: the reference model transfers unchanged from lists to instances.

Wrong / right grid

What teams will write (wrong)What happensThe fix (right)
def print_status(): — no selfTypeError: print_status() takes 0 positional arguments but 1 was givendef print_status(self): — the instance is auto-inserted as arg 1
c1.print_status — no parenthesesPrints a bound-method object; no output linec1.print_status() — call it
number = "" inside __init__ without self.Creates a local variable that vanishes; later c1.number raises AttributeErrorself.number = ""
Predicting c1 still shows RiveraTeam believes c3 = c1 copied the cabinAliasing: two names, one object — same as b = a with lists

Bonus extensions (fast teams)

  • Add a check_out(self) that resets guest to "Vacant".
  • Give check_in a return value (True/False) instead of printing — discuss why returning beats printing for reusable code.
  • Preview of 10.5: "Wouldn't it be nicer to write Cabin("A101", 7)? Hold that thought for twenty minutes."

Debrief script (3 min): Poll in chat first: "After c3.guest = "Dubois", what does c1 show?" Expect a 50/50 split — that split is the lesson. Then run it live in Python Tutor if time allows: two variables, one arrow. Close by connecting to the deck's alias-trap callout. Watch for teams that never ran the code and answered from vibes — call for the actual output line.

Key takeaway to say out loud: "Instances are objects like lists are objects. Assignment never copies — it points."

Breakout 2 · ~7:05 PM · 10 minutes

🎟️ Shore Excursion Bookings

Problem recap: Class attribute pier shared by all bookings; constructor with two required and two defaulted parameters (party_size=2, rate=79.00); a receipt method; three instantiations exercising defaults and a by-name argument; then mutate Excursion.pier and observe all receipts change.

Base solution — annotated

excursion_solution.py
class Excursion:
    pier = "Terminal J"                          # CLASS attribute - one shared copy

    def __init__(self, guest, tour, party_size=2, rate=79.00):
        self.guest = guest                       # required, positional, in order
        self.tour = tour
        self.party_size = party_size             # defaults kick in if omitted
        self.rate = rate

    def print_receipt(self):
        total = self.party_size * self.rate      # local variable, dies at return
        print(f"{self.guest} - {self.tour} x{self.party_size} "
              f"from {self.pier}: ${total:.2f}") # self.pier falls through to class

b1 = Excursion("Okafor", "Snorkel Adventure", 3)     # rate defaults to 79.00
b2 = Excursion("Chen", "Old Town Walk")              # both defaults
b3 = Excursion("Rivera", "Reef Dive", rate=129.00)   # by NAME: party_size stays 2

for b in [b1, b2, b3]:
    b.print_receipt()

print("--- pier change ---")
Excursion.pier = "Terminal F"                    # turn the ONE shared dial
for b in [b1, b2, b3]:
    b.print_receipt()
Okafor - Snorkel Adventure x3 from Terminal J: $237.00 Chen - Old Town Walk x2 from Terminal J: $158.00 Rivera - Reef Dive x2 from Terminal J: $258.00 --- pier change --- Okafor - Snorkel Adventure x3 from Terminal F: $237.00 Chen - Old Town Walk x2 from Terminal F: $158.00 Rivera - Reef Dive x2 from Terminal F: $258.00

Task 5 — expected answer

All three receipt lines change pier. pier lives in the class namespace; every instance's self.pier lookup misses in the instance namespace and falls through to the class — one dial, all instances.

Wrong / right grid

What teams will write (wrong)What happensThe fix (right)
Excursion("Rivera", "Reef Dive", 129.00)129.00 lands in party_size, rate stays 79.00 — receipt reads x129.0 with total $10191.00. Runs without error; watch how many teams don't noticePass by name: rate=129.00 — order-free and self-documenting
b1.pier = "Terminal F" for task 5Creates an instance attribute on b1 that shadows the class attribute — only b1's receipt changesExcursion.pier = "Terminal F" — mutate the class namespace
self.pier = "Terminal J" inside __init__Runs, but now every instance owns a private copy — task 5 then changes nothing, contradicting the lessonDefine pier in class scope, outside all methods
${total:.2} or round(total)Wrong decimals ("$2.4e+02"-style surprises with :.2){total:.2f} — the f matters

Bonus extensions (fast teams)

  • Add a class attribute port_fee = 5.00 per person and fold it into the total.
  • Ask: "What prints for b1.pier if you did the b1.pier shadow-write, then reset with del b1.pier?" (Lookup falls back to class again — nice namespace proof.)

Debrief script (3 min): The gold is the positional 129.00 bug — at least one team will have a $10k+ snorkel trip and not blink. Put their (anonymized) receipt on screen and ask the room what's wrong before revealing. Then show the shadow-write variant (b1.pier = ...) in 30 seconds: instance namespace wins, class untouched. That's Quick Check 2 knowledge cashing out.

Key takeaway to say out loud: "Defaults answer 'what's typical?' Class attributes answer 'what's true for everyone?' Instance attributes answer 'what's true for this one?'"

Breakout 3 · ~7:35 PM · 8 minutes

🍎 Galley Inventory — derive it

Problem recap: Derive Perishable from the deck's Item class; call Item.__init__(self); add expiration and display_full() that reuses the inherited display(); instantiate one of each; answer the Dwelling/House method-count question.

Base solution — annotated

galley_solution.py
class Item:                                    # base class (from the slide)
    def __init__(self):
        self.name = ""
        self.quantity = 0
    def set_name(self, nm):
        self.name = nm
    def set_quantity(self, qnty):
        self.quantity = qnty
    def display(self):
        print(self.name, self.quantity)

class Perishable(Item):                        # derived: inherits Item's methods
    def __init__(self):
        Item.__init__(self)                    # REQUIRED: builds name/quantity
        self.expiration = ""                   # then add the new attribute
    def set_expiration(self, expir):
        self.expiration = expir
    def display_full(self):
        self.display()                         # reuse the inherited method
        print(f"  (Expires: {self.expiration})")

dry = Item()
dry.set_name("Rice")
dry.set_quantity(120)
dry.display()

cold = Perishable()
cold.set_name("Mangoes")                       # inherited from Item
cold.set_quantity(60)                          # inherited from Item
cold.set_expiration("Jul 24, 2026")
cold.display_full()
Rice 120 Mangoes 60 (Expires: Jul 24, 2026)

Task 4 — expected answer

3 methods: open_door() (inherited from Dwelling) + open_window() + open_basement(). This is verbatim zyBooks 10.6.4 — it will feel like free points to teams who did the reading, which is exactly the message.

Wrong / right grid

What teams will write (wrong)What happensThe fix (right)
Omitting Item.__init__(self)AttributeError: 'Perishable' object has no attribute 'name' the moment display() runs — instance attributes are not inherited automaticallyCall the base constructor first line of the derived constructor
Item.__init__() — forgot to pass selfTypeError: __init__() missing 1 required positional argument: 'self' — explicit base-class calls do NOT auto-insert the instanceItem.__init__(self)
self.display(self) in display_fullTypeError: display() takes 1 positional argument but 2 were given — the mirror image of the forgot-self bugself.display() — dot call auto-inserts self
Task 4 answer: "2"Team forgot inherited methods countDerived instances can call base methods: 1 + 2 = 3

Bonus extensions (fast teams)

  • Derive Frozen(Perishable) adding freezer_deck — a two-level tree, mirroring Item → Produce → Fruit from the slide.
  • Pythonic detour: replace the setters with direct attribute writes (cold.name = "Mangoes") and discuss why zyBooks' getter/setter style is a Java accent, not Python's.

Debrief script (2–3 min): Run the broken no-base-constructor version live first — the AttributeError is the teachable crash, and it lands better as a demo than a warning. Trace it in Python Tutor Demo 2 if the room looks shaky. Collect task-4 numbers in chat before revealing; wrong answers of "2" tell you who thinks inheritance only flows to new methods. Keep this one tight — it's the last breakout before break and the room will be fading.

Key takeaway to say out loud: "Methods are inherited for free. Instance attributes are not — you must invite the base constructor to the party."

Breakout 4 · ~8:50 PM · 10 minutes

🌡️ Lord Kelvin gets modularized

Problem recap: Refactor last week's converter: math in temp_tools.py, UI in main.py. Add an unguarded self-test to the module, observe the import side effect, then fix it with the __name__ == "__main__" guard. Stretch: switch to from-imports.

Base solution — annotated

temp_tools.py (final, guarded)
# temp_tools.py - reusable conversion module
def c_to_f(c):
    return c * 9 / 5 + 32

def c_to_k(c):
    return c + 273.15

if __name__ == "__main__":          # True ONLY when run directly
    # self-test: fires on  python temp_tools.py,  silent on import
    print(c_to_f(100))              # expect 212.0
    print(c_to_k(0))                # expect 273.15
main.py
# main.py - user interface script
import temp_tools                   # executes the module ONCE (sys.modules)

c = float(input("Enter Celsius: "))
print(f"{temp_tools.c_to_f(c):.2f} F")
print(f"{temp_tools.c_to_k(c):.2f} K")
Enter Celsius: 25 77.00 F 298.15 K

Task 3 — expected answer

With the unguarded self-test, running main.py prints 212.0 and 273.15 before the input prompt even appears. Why: import temp_tools executes every top-level statement in the module — imports are execution, not inclusion. The guard makes the self-test conditional on being the executing script (__name__ is "__main__") rather than an import (__name__ is "temp_tools").

Wrong / right grid

What teams will write (wrong)What happensThe fix (right)
import temp_tools.pyModuleNotFoundError: No module named 'temp_tools.py' — the .py reads as a dotted package pathimport temp_tools — no extension, ever
print(c_to_f(c)) after plain import temp_toolsNameError: name 'c_to_f' is not defined — plain import adds only the module namePrefix it (temp_tools.c_to_f) or use the stretch's from-import
if __name__ == "main": or '__main__' with one underscoreCondition silently never True — self-test never runs, team declares victory for the wrong reasonExactly "__main__": two underscores each side. Have them run temp_tools.py directly to prove the test still fires
Files in different foldersModuleNotFoundError despite perfect code — sys.path rule 2a: script's own directoryKeep module and script side by side (tonight's 11.5 slide)

Bonus extensions (fast teams)

  • Add f_to_c(f) and re-run only the module's self-test — no changes needed in main.py: that's the maintainability win, name it.
  • Ask them to predict: does importing temp_tools twice print the self-test twice with the guard removed? (No — sys.modules caches; step 1 of the import process.)

Debrief script (4 min): Task 3 is the whole activity — collect one-sentence "why" answers in chat and read two aloud; you want the phrase "import runs the file" from a student's mouth, not yours. Verify at least one team demos both runs (main.py silent, temp_tools.py chatty) — that two-sided test is the comprehension check, and it's un-fakeable with AI because it requires running code in front of you. Land the DNA-project connection hard: "Your DNA search functions go in a module. Your script gets this guard. I will be looking for it."

Key takeaway to say out loud: "A module is a program that got promoted. The guard is how it stays polite when other programs hire it."
