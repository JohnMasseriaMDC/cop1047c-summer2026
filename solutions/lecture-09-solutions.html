Breakout Activity #1

🧩 Grocery Bill Splitter — try / except in Action

🧩
Breakout #1  ·  12 minutes
Grocery Bill Splitter
try / except ZeroDivisionError raise while loop
Problem Recap
  • Read total_bill (float) and num_people (int) from input
  • Compute and print total_bill / num_people, formatted to 2 decimals
  • Catch ValueError — print "Please enter valid numbers."
  • Catch ZeroDivisionError — print "Cannot split among zero people."
  • Repeat in a while loop until the user enters "q"
  • Bonus A: add a 20% tip before dividing
  • Bonus B: catch a negative num_people by raising a ValueError yourself
1Base Solution
breakout_grocery_bill.py 
while True:
    total_bill_input = input("Enter total bill (q to quit): ")
    if total_bill_input.lower() == "q":
        break

    try:
        total_bill = float(total_bill_input)
        num_people = int(input("Enter number of people: "))
        share = total_bill / num_people
        print(f"Each person owes: ${share:.2f}")
    except ValueError:
        print("Please enter valid numbers.")
    except ZeroDivisionError:
        print("Cannot split among zero people.")

print("Goodbye!")
Sample run
Enter total bill (q to quit): 84.00
Enter number of people: 4
Each person owes: $21.00

Enter total bill (q to quit): seventy
Please enter valid numbers.

Enter total bill (q to quit): 50
Enter number of people: 0
Cannot split among zero people.

Enter total bill (q to quit): q
Goodbye!
2Line-by-Line Explanation
  • Line 2 total_bill_input = input(...) — read as a string first, before any conversion. This is what lets us check for "q" safely — if we converted to float() immediately, typing "q" would crash before we ever got the chance to check for it.
  • Lines 6–9 Two conversions, one try block — float(total_bill_input) and int(input(...)) can each raise ValueError independently. Since both should be handled the same way, one shared try covering both is cleaner than two separate ones.
  • Line 10 total_bill / num_people — this is where ZeroDivisionError can occur, but only after both conversions already succeeded. Python doesn't know which line will fail until it runs — that's exactly why we let one try block cover the whole sequence.
  • Lines 12–15 Two separate except blocks — each one names a specific exception type and gives a message tailored to that exact problem, instead of one vague catchall.
Why check for "q" as a string? Converting user input happens after we've already decided the user wants to keep going. Sentinel checks like this should always happen on the raw string, before any int() or float() conversion.
3Common Mistakes
❌ Converted before checking for "q"
total_bill = float(input("Enter total bill (q to quit): "))
if total_bill == "q":
    break
# ValueError crashes the program the moment
# the user types "q" -- float("q") fails
# before the if-statement ever runs.
✅ Check the raw string first
total_bill_input = input("Enter total bill (q to quit): ")
if total_bill_input.lower() == "q":
    break
# Only convert AFTER we know it's not the
# sentinel value.
total_bill = float(total_bill_input)
❌ One bare except: for everything
try:
    total_bill = float(total_bill_input)
    num_people = int(input("Enter number of people: "))
    share = total_bill / num_people
    print(f"Each person owes: ${share:.2f}")
except:
    print("Something went wrong.")
# The user has no idea WHICH thing went wrong.
✅ Specific handlers, specific messages
except ValueError:
    print("Please enter valid numbers.")
except ZeroDivisionError:
    print("Cannot split among zero people.")
# Each message tells the user exactly
# what to fix.
⭐ Bonus A — Add a 20% Tip Before Dividing

This is pure arithmetic — no new exception-handling concepts — but it has to happen inside the try block, since it depends on total_bill having already converted successfully.

breakout_grocery_bonus_a.py 
try:
    total_bill = float(total_bill_input)
    tip = total_bill * 0.20
    total_with_tip = total_bill + tip
    num_people = int(input("Enter number of people: "))
    share = total_with_tip / num_people
    print(f"Each person owes (with tip): ${share:.2f}")
except ValueError:
    print("Please enter valid numbers.")
except ZeroDivisionError:
    print("Cannot split among zero people.")
Sample run — total_bill = 84.00, num_people = 4
Each person owes (with tip): $25.20
⭐ Bonus B — Reject a Negative Number of People

Python's int() happily converts "-3" to -3 — it's a perfectly valid integer, just not a valid number of people. This is a business rule, not a type error, so we raise our own ValueError to enforce it.

breakout_grocery_bonus_b.py 
try:
    total_bill = float(total_bill_input)
    num_people = int(input("Enter number of people: "))
    if num_people < 0:
        raise ValueError("Number of people cannot be negative.")
    share = total_bill / num_people
    print(f"Each person owes: ${share:.2f}")
except ValueError as excpt:
    print(excpt)
except ZeroDivisionError:
    print("Cannot split among zero people.")
Test cases
total_bill=50, num_people=-2  -> Number of people cannot be negative.
total_bill=fifty              -> invalid literal for int() with base 10: 'fifty'
total_bill=50, num_people=2   -> Each person owes: $25.00
Notice: both our own raise and Python's own conversion failures land in the same except ValueError as excpt block. Whatever message the exception was created with — ours or Python's — is what print(excpt) shows.
4Both Bonuses Together
breakout_grocery_complete.py 
while True:
    total_bill_input = input("Enter total bill (q to quit): ")
    if total_bill_input.lower() == "q":
        break

    try:
        total_bill = float(total_bill_input)
        num_people = int(input("Enter number of people: "))

        if num_people < 0:
            raise ValueError("Number of people cannot be negative.")

        tip = total_bill * 0.20
        total_with_tip = total_bill + tip
        share = total_with_tip / num_people
        print(f"Each person owes (with tip): ${share:.2f}")
    except ValueError as excpt:
        print(excpt)
    except ZeroDivisionError:
        print("Cannot split among zero people.")

print("Goodbye!")
Key Takeaway — Breakout #1

Not every "error" comes from Python itself — raise lets you enforce your own business rules (like "people can't be negative") using the exact same try / except machinery that catches built-in errors. Whether an exception was raised by int() failing or by your own raise statement, the calling code doesn't need to know or care — it just catches ValueError either way.

Breakout Activity #2

🧩 Grade Validator — Custom Exceptions

🧩
Breakout #2  ·  15 minutes
Grade Validator
custom exception raise functions propagation
Problem Recap
  • Define class InvalidGradeError(Exception): with a constructor that stores the bad value
  • Write get_grade() — reads/validates an integer grade, raise InvalidGradeError(...) if outside 0–100
  • In a while loop, call get_grade() inside a try block
  • Catch InvalidGradeError and print its message
  • On success, print the letter grade (A/B/C/D/F) using the syllabus grade scale
  • Loop until the user enters "q"
  • Bonus: also catch plain ValueError for non-numeric input, separately from InvalidGradeError
1Base Solution
breakout_grade_validator.py 
class InvalidGradeError(Exception):
    '''Raised when a grade is outside the valid 0-100 range.'''
    def __init__(self, value):
        self.value = value
        super().__init__(f"Invalid grade: {value} is not between 0 and 100.")


def get_grade(raw_input):
    grade = int(raw_input)               # may raise ValueError
    if grade < 0 or grade > 100:
        raise InvalidGradeError(grade)
    return grade


def letter_for(grade):
    if grade >= 90:
        return "A"
    elif grade >= 80:
        return "B"
    elif grade >= 70:
        return "C"
    elif grade >= 60:
        return "D"
    else:
        return "F"


while True:
    raw = input("Enter grade (q to quit): ")
    if raw.lower() == "q":
        break

    try:
        grade = get_grade(raw)
        print(f"Grade: {letter_for(grade)}")
    except InvalidGradeError as excpt:
        print(excpt)
Sample run
Enter grade (q to quit): 92
Grade: A

Enter grade (q to quit): 150
Invalid grade: 150 is not between 0 and 100.

Enter grade (q to quit): abc
Traceback (most recent call last):
  ...
ValueError: invalid literal for int() with base 10: 'abc'
    -- crashes! This is exactly what the Bonus below fixes.
2Line-by-Line Explanation
  • Lines 1–5 class InvalidGradeError(Exception): — inherits from the built-in Exception class, per 8.6. The constructor stores the bad value as an attribute, then calls super().__init__(...) to set the message that print(excpt) will display.
  • Line 9 grade = int(raw_input) — this line can still raise a plain ValueError if raw_input isn't numeric at all. get_grade() doesn't catch it — it just lets it propagate to whoever calls the function (8.4!).
  • Lines 10–11 The range check — only runs if int() already succeeded. If the grade is out of range, we raise InvalidGradeError(grade) instead of returning — the function exits immediately.
  • Lines 33–36 try: grade = get_grade(raw) — because get_grade() has no try of its own, both possible exceptions (ValueError from int(), or our own InvalidGradeError) climb the call stack to this try block — the same propagation idea from 8.4's get_weight() / get_height() example.
Why does the base solution crash on "abc"? Because it only has except InvalidGradeError — there's no handler for the plain ValueError that int("abc") raises. That gap is exactly what the Bonus below closes.
3Common Mistakes
❌ Forgot super().__init__(...)
class InvalidGradeError(Exception):
    def __init__(self, value):
        self.value = value
        # no super().__init__() call!

# print(excpt) now prints an EMPTY string --
# Exception's default message is blank unless
# you set one.
✅ Call super().__init__(message)
class InvalidGradeError(Exception):
    def __init__(self, value):
        self.value = value
        super().__init__(
            f"Invalid grade: {value} is not between 0 and 100."
        )
# Now print(excpt) shows this exact message.
❌ Reading "q" through int(input()) directly
while True:
    try:
        grade = get_grade(input("Enter grade (q to quit): "))
    except InvalidGradeError as excpt:
        print(excpt)
# Typing "q" now raises a plain ValueError that
# ISN'T caught here -- the loop crashes instead
# of quitting cleanly.
✅ Check the sentinel before converting
while True:
    raw = input("Enter grade (q to quit): ")
    if raw.lower() == "q":
        break
    try:
        grade = get_grade(raw)
    except InvalidGradeError as excpt:
        print(excpt)
⭐ Bonus — Catch Plain ValueError Too

Add a second, more general except block after the specific one. Since InvalidGradeError inherits from Exception (not ValueError), the two are independent — order between them doesn't matter here. It would matter if InvalidGradeError had been defined as a subclass of ValueError instead — then the more specific type must come first, or the general handler would catch it first and the specific one would never run.

breakout_grade_validator_bonus.py 
while True:
    raw = input("Enter grade (q to quit): ")
    if raw.lower() == "q":
        break

    try:
        grade = get_grade(raw)
        print(f"Grade: {letter_for(grade)}")
    except InvalidGradeError as excpt:
        print(excpt)
    except ValueError:
        print("Please enter a whole number.")
Sample run
Enter grade (q to quit): 92
Grade: A

Enter grade (q to quit): 150
Invalid grade: 150 is not between 0 and 100.

Enter grade (q to quit): abc
Please enter a whole number.

Enter grade (q to quit): q
This now matches the full spec exactly: a valid grade prints a letter, an out-of-range grade prints our custom message, non-numeric input prints a friendly message instead of crashing, and "q" exits cleanly.
Key Takeaway — Breakout #2

get_grade() intentionally has no try block of its own — it's the caller's job to decide how to react to a bad grade. This is the propagation idea from 8.4 in action: a function can raise (or let propagate) an exception without knowing or caring how it will eventually be handled. Keeping validation logic and error-handling logic in separate places is exactly why exceptions exist.

Breakout Activity #3

🧩 Attendance Log — Reading & Writing Text Files

🧩
Breakout #3  ·  12 minutes
Attendance Log
append mode write() / readlines() try / finally
Problem Recap
  • Open "attendance.txt" in "a" mode
  • Read a student name from input, write() it followed by "\n"
  • Loop until the user types "done", then close() the file
  • Re-open in "r" mode, use readlines(), and print the total number of names logged
  • Wrap the whole thing in try / except to catch ValueError / IOError gracefully
  • Bonus: use try / finally to guarantee the write file always closes, even if something crashes mid-loop
1Base Solution
breakout_attendance_log.py 
try:
    write_file = open("attendance.txt", "a")

    while True:
        name = input("Enter name (done to finish): ")
        if name.lower() == "done":
            break
        write_file.write(name + "\n")

    write_file.close()

    read_file = open("attendance.txt", "r")
    lines = read_file.readlines()
    read_file.close()

    print(f"\nAttendance recorded: {len(lines)} students")
except IOError:
    print("Could not open attendance.txt.")
except ValueError:
    print("Unexpected input error.")
Sample run (fresh attendance.txt)
Enter name (done to finish): Ana
Enter name (done to finish): Beto
Enter name (done to finish): done

Attendance recorded: 2 students
2Line-by-Line Explanation
  • Line 2 open("attendance.txt", "a") — append mode creates the file if it's missing, and adds new writes to the end of whatever is already there.
  • Line 8 write_file.write(name + "\n") — write() does not add a newline automatically. Forget the "\n" and every name runs together on one line.
  • Lines 11–13 Close, then re-open — we must close() the append-mode file before reopening the same file for reading, so any buffered writes are flushed to disk first (recall 12.2's output buffering).
  • Line 15 len(lines) — readlines() returns one list element per line, so its length is exactly the number of names logged all-time, not just this run.
Cumulative by design: because we use "a" mode, running this program multiple times keeps adding to attendance.txt. The count reported reflects everyone ever logged, across every run — delete the file first if you want a clean demo.
3Common Mistakes
❌ Used "w" instead of "a"
write_file = open("attendance.txt", "w")
# Every run ERASES all previous names
# before writing new ones.
✅ "a" preserves history
write_file = open("attendance.txt", "a")
# New names are appended -- previous
# sessions' names stay intact.
❌ Forgot the newline
write_file.write(name)
# "Ana" + "Beto" + "Cleo" all run together
# on one line: "AnaBetoCleo"
# readlines() later returns ONE element,
# not three.
✅ Add "\n" after every write
write_file.write(name + "\n")
# Each name lands on its own line, so
# readlines() correctly returns one
# element per student.
⭐ Bonus — Guarantee the Write File Closes

If input() hits an unexpected end-of-file, or any other error interrupts the loop, the base solution's write_file.close() on line 9 would never run — leaving the file open. Wrapping the loop in try / finally guarantees the close happens no matter what.

breakout_attendance_log_bonus.py 
write_file = open("attendance.txt", "a")
try:
    while True:
        name = input("Enter name (done to finish): ")
        if name.lower() == "done":
            break
        write_file.write(name + "\n")
finally:
    write_file.close()
    print("Write file closed.")

with open("attendance.txt", "r") as read_file:
    lines = read_file.readlines()

print(f"\nAttendance recorded: {len(lines)} students")
Sample run
Enter name (done to finish): Ana
Enter name (done to finish): Beto
Enter name (done to finish): done
Write file closed.

Attendance recorded: 2 students
Nice preview: the read side already uses with, which you just learned in 12.6 — it's the modern replacement for manual try / finally cleanup, and you'll default to it for nearly every file you open from here on.
Key Takeaway — Breakout #3

Mode choice isn't a small detail — "a" vs. "w" is the difference between a growing log and a file that resets every run. And a resource like an open file needs a guaranteed cleanup path: finally (or, better still, with) makes sure that happens even when something goes wrong mid-loop.

Breakout Activity #4

🧩 Safe File Copier — with + Exception Handling

🧩
Breakout #4  ·  12 minutes
Safe File Copier
os.path.exists with statement os.path.join
Problem Recap
  • Read a source filename from input
  • Use os.path.exists() to check it first — if missing, print an error and stop
  • Use two with statements (or one, comma-separated) to open the source in "r" mode and a new file "copy_of_" + filename in "w" mode
  • Loop for line in source: and write() each line to the new file
  • Print how many lines were copied
  • Wrap any numeric-summing step in try / except to catch ValueError
  • Bonus: use os.path.join() to build the output path instead of string concatenation
1Base Solution
breakout_safe_file_copier.py 
import os

filename = input("Enter file to copy: ")

if not os.path.exists(filename):
    print("File does not exist.")
else:
    output_name = "copy_of_" + filename
    line_count = 0
    total = 0

    with open(filename, "r") as source, open(output_name, "w") as destination:
        for line in source:
            destination.write(line)
            line_count += 1
            try:
                total += int(line)
            except ValueError:
                pass   # not every line has to be a number

    print(f"Copied {line_count} lines to {output_name}")
    print(f"Sum of numeric lines: {total}")
Sample run
Enter file to copy: notes.txt
Copied 14 lines to copy_of_notes.txt
Sum of numeric lines: 245

Enter file to copy: ghost.txt
File does not exist.
2Line-by-Line Explanation
  • Line 5 os.path.exists(filename) — check before opening. This avoids ever calling open() on a path that isn't there, and lets us print a friendly message and stop cleanly.
  • Line 12 with open(filename, "r") as source, open(output_name, "w") as destination: — a single with statement can open multiple files, separated by commas. Both are guaranteed to close together when the block ends.
  • Lines 13–18 One pass, two jobs — each line is written to the copy and optionally added to a running total, all in a single for loop. Reading the file twice would be wasteful.
  • Lines 16–18 A nested try / except — the inner try only wraps the int(line) conversion. A line that isn't numeric simply doesn't add to total; it doesn't stop the copy from continuing.
Why check os.path.exists() instead of just wrapping open() in a try? Either approach works here. Checking first reads a little more naturally when you plan to print one specific message and stop; catching FileNotFoundError around open() (recall the Chapter 8 tie-in from Lecture 9's deck) is the better choice once multiple things could go wrong with the open itself.
3Common Mistakes
❌ Skipped the existence check
filename = input("Enter file to copy: ")
with open(filename, "r") as source:
    ...
# Typo the filename and you get a raw
# FileNotFoundError traceback instead of
# a friendly message.
✅ Check first, act second
if not os.path.exists(filename):
    print("File does not exist.")
else:
    with open(filename, "r") as source:
        ...
❌ Reading the whole file twice
for line in source:
    destination.write(line)
for line in source:          # source is already exhausted!
    total += int(line)        # this loop does nothing at all
✅ One loop, both jobs
for line in source:
    destination.write(line)
    try:
        total += int(line)
    except ValueError:
        pass
⭐ Bonus — Build the Output Path with os.path.join()

String concatenation ("copy_of_" + filename) works fine when everything lives in the current directory. The moment you want the copy somewhere else — like a "backups" folder — os.path.join() keeps the path portable across operating systems.

breakout_safe_file_copier_bonus.py 
import os

filename = input("Enter file to copy: ")

if not os.path.exists(filename):
    print("File does not exist.")
else:
    # os.path.join builds a portable path -- correct
    # separator on Windows ("\") vs. Mac/Linux ("/")
    output_name = os.path.join(".", "copy_of_" + filename)
    line_count = 0

    with open(filename, "r") as source, open(output_name, "w") as destination:
        for line in source:
            destination.write(line)
            line_count += 1

    print(f"Copied {line_count} lines to {output_name}")
Output (Mac/Linux vs. Windows)
Mac/Linux:  Copied 14 lines to ./copy_of_notes.txt
Windows:    Copied 14 lines to .\copy_of_notes.txt
Same code, different separator — os.path.join() picked the right one automatically for whichever OS ran the program. This becomes essential the moment you introduce real subdirectories, like os.path.join("backups", "copy_of_" + filename).
Key Takeaway — Breakout #4

A safe file operation checks its assumptions before acting (os.path.exists()), guarantees cleanup regardless of outcome (with), and never hardcodes something the operating system should decide for you (os.path.join() over string concatenation). These three habits will show up in almost every file-handling program you write from here on — including the course's final project.
