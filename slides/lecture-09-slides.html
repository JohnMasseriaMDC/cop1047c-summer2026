8.1 — Introduction

Errors Happen: Why We Need Exception Handling

Exception

An exception is an error detected during execution. Python has special try / except constructs specifically to handle these exceptional circumstances.

  • An unhandled exception immediately terminates the program
  • Python prints a Traceback showing where the error occurred
  • Good programs anticipate bad input and keep running instead of crashing
Real talk: a user will eventually type something you didn't expect. The question isn't if, it's when. 
weight = int(input("Enter weight: "))
height = int(input("Enter height: "))
bmi = (float(weight) / float(height * height)) * 703
print(f"BMI: {bmi}")

# --- if the user types letters instead ---
# Enter weight: One-hundred fifty
#
# Traceback (most recent call last):
#   File "test.py", line 1, in <module>
#     weight = int(input("Enter weight: "))
# ValueError: invalid literal for int()
#             with base 10: 'One-hundred fifty'
#
# Program terminates. Nothing after this line runs.
8.1 — Basic Construct

The try / except Construct

try:
    weight = int(input("Enter weight: "))
    height = int(input("Enter height: "))
    bmi = (float(weight) / float(height * height)) * 703
    print(f"BMI: {bmi}")
except:
    print("Could not calculate health info.")

print("Program keeps going!")
Control Flow

Statements in the try block run first. If no exception occurs, except is skipped entirely.

  • If an exception does occur, execution jumps immediately to except
  • Any remaining try statements are skipped — they never run
  • After except finishes, the program continues normally afterward
Mental model: "try this — if anything goes wrong anywhere in here, jump down there and recover."

8.2 — Multiple Handlers

Different Errors, Different Handlers

try:
    weight = int(input("Enter weight: "))
    height = int(input("Enter height: "))
    bmi = (float(weight) / float(height * height)) * 703
    print(f"BMI: {bmi}")
except ValueError:
    print("Please enter whole numbers only.")
except ZeroDivisionError:
    print("Invalid height entered. Must be > 0.")
Why Multiple except Blocks?

A single try block can raise different kinds of errors. Each except can target one specific type with its own recovery message.

  • Python checks each except top to bottom and runs the first match
  • Only one handler runs per exception — then execution resumes after the whole block
  • An unspecified exception type with no matching handler is left unhandled — program still crashes
8.2 — Grouping Types & Best Practice

One Handler, Several Types — And Why except: Alone Is Risky

try:
    result = my_lib.magic()
    f = open(result, "r")
    print(f.read())
except (ValueError, TypeError):
    print("Bad input or bad type.")
except (NameError, AttributeError):
    print("That function doesn't exist.")
except:
    print("Something else went wrong.")
  • A tuple like (ValueError, TypeError) lets one block handle several types identically
  • An except: with no type is a catchall — it handles any exception
Good practice: avoid a bare catchall when you can. It can hide real bugs by silently swallowing errors you never anticipated — always prefer naming the specific exception type(s) you expect.

8.3 — The raise Statement

Raising Your Own Exceptions Keeps Code Clean

Why raise Instead of if / else?

Sprinkling if / else checks throughout "normal" code obscures the program's flow. A raise statement exits the try block immediately and jumps to the handler.

  • raise ValueError("message") creates a new exception with a descriptive string
  • You can raise any built-in type — pick the one that best fits the problem
  • The normal code flow reads cleanly: "get weight, get height, print BMI" — not buried in branches
try:
    weight = int(input("Enter weight: "))
    if weight < 0:
        raise ValueError("Invalid weight.")

    height = int(input("Enter height: "))
    if height <= 0:
        raise ValueError("Invalid height.")

    bmi = (float(weight) * 703) / (height * height)
    print(f"BMI: {bmi}")
except ValueError as excpt:
    print(excpt)
    print("Could not calculate health info.")
8.3 — except ... as

except ValueError as excpt — Inspecting the Error

try:
    user_age = int(input())
    if user_age < 0:
        raise ValueError("Invalid age")
    avg_max_heart_rate = 220 - user_age
    print(f"Avg: {avg_max_heart_rate}")
except ValueError as excpt:
    print(f"Error: {excpt}")

# Input: 50   -> Avg: 170
# Input: -5   -> Error: Invalid age
The as Keyword

The keyword as binds a name to the exception object that was raised. Printing that name prints the message string passed to raise.

  • Works the same whether the exception came from raise or from Python itself (like int() failing)
  • Great for giving the user a specific, useful message instead of a generic one
8.4 — Functions & the Call Stack

An Unhandled Exception Climbs the Call Stack

def get_weight():
    weight = int(input("Enter weight: "))
    if weight < 0:
        raise ValueError("Invalid weight.")
    return weight

def get_height():
    height = int(input("Enter height: "))
    if height <= 0:
        raise ValueError("Invalid height.")
    return height

try:
    weight = get_weight()
    height = get_height()
    print(f"BMI: {(weight/(height*height))*703}")
except ValueError as excpt:
    print(excpt)
Propagation

If get_weight() raises an exception and has no try block of its own, the function exits immediately. Python looks up the call stack for a handler.

  • The exception is caught wherever the call to get_weight() happens to be wrapped in try
  • Without exceptions, functions would need a special "failure" return value (like -1) — and the caller would need extra if/else to check it every time
A key goal of exception handling: keep normal code readable, free of scattered error-checking branches.
8.5 — The finally Clause

finally Always Runs

def divide(a, b):
    z = -1
    try:
        z = a / b
    except ZeroDivisionError:
        print("Cannot divide by zero")
    finally:
        print(f"Result is {z}")

divide(4, 2)   # Result is 2.0
divide(4, 0)   # Cannot divide by zero
               # Result is -1
  • No exception? finally runs right before the try block finishes
  • Handled exception? the except block runs, then finally runs
  • Unhandled exception? finally still runs — then the exception is re-raised
  • break, continue, or return inside the try also trigger finally before exiting
Use case: clean-up code that must run no matter what — like closing a file, releasing a resource, or printing a status line.
8.5 — A Realistic Example

Closing a File No Matter What Happens

nums = []
my_file = input("Enter file name: ")
rd_nums = -1
try:
    rd_nums = open(my_file, "r")
    for line in rd_nums:
        nums.append(int(line))
except IOError:
    print(f"Could not find {my_file}")
except ValueError:
    print(f"Could not read number from {my_file}")
finally:
    print(f"Closing {my_file}")
    if rd_nums != -1:
        rd_nums.close()
  • Whether the file is missing, contains bad data, or reads perfectly — finally guarantees the close happens
  • Forgetting to close a file can leave it locked so other programs can't use it

8.6 — Building Your Own Error Types

Custom Exception Types

# Define a custom exception type
class LessThanZeroError(Exception):
    def __init__(self, value):
        self.value = value

my_num = int(input("Enter number: "))
if my_num < 0:
    raise LessThanZeroError("my_num must be > 0")
else:
    print(f"my_num: {my_num}")
Why Build Your Own?

Built-in types like ValueError are generic. A custom type names the exact problem your program cares about.

  • Create it by inheriting from the built-in Exception class
  • Keep it minimal — a constructor that stores an attribute, or just pass
  • Good practice: end the class name with Error — e.g. LessThanZeroError, InvalidGradeError

Applied — Debugging Habits

Common Pitfalls to Avoid

Bare except: hides bugs. A catchall silently swallows any error, including ones you never intended to handle — like a typo in a variable name. Name the specific type(s) you expect.
Don't use exceptions for normal control flow. If you can easily check a condition with if beforehand (like if height > 0), that's often clearer than relying on a crash to tell you.
Write informative messages. raise ValueError("Invalid weight.") is far more useful to a future debugger (probably you!) than a bare raise ValueError().
Order matters. Put more specific except blocks before more general ones — Python uses the first match it finds, top to bottom.

12.1 — open() and read()

Reading From a File

The File Object

open("myfile.txt") creates a file object connected to that file on disk. Reading input this way replaces keyboard entry with data already saved somewhere.

  • f.read() returns the entire file as one string
  • f.close() releases the file — always do this when finished
  • Path can be relative ("myfile.txt") or absolute ("C:\\Users\\...)
print("Opening file myfile.txt.")
f = open("myfile.txt")        # create file object
print("Reading file myfile.txt.")
contents = f.read()           # read entire file as a string
print("Closing file myfile.txt.")
f.close()                     # close the file

print("\nContents of myfile.txt:")
print(contents)

# but not the one it needs right now.
12.1 — Three Ways to Read

read() vs. readlines() vs. Iterating a File

# read() -> one big string
f = open("myfile.txt")
contents = f.read()

# readlines() -> a list, one element per line
f2 = open("myfile.txt")
lines = f2.readlines()
print(lines[1])          # second line

# Iterate directly over the file object
f3 = open("myfile.txt")
for line in f3:
    print(line, end="")  # each line already has \n
f3.close()
  • read() → one string with the whole file
  • readlines() → a list of strings, one per line (each keeps its trailing \n)
  • readline() → a single line at a time — useful for huge files that won't fit in memory
  • File objects support for line in f: directly — the most common pattern
All three stop automatically at end-of-file (EOF) — no more data means the loop or read simply ends.
12.1 — A Common Pattern

Read → Iterate → Compute

# mydata.txt has 100 integers, one per line
f = open("mydata.txt")
lines = f.readlines()
f.close()

total = 0
for ln in lines:
    total += int(ln)   # must convert string -> int

avg = total / len(lines)
print(f"Average value: {avg}")
The Three-Step Pattern

1) Read the file's contents. 2) Iterate over each line to process values. 3) Compute a result, such as a total or average.

Remember: everything read from a file is a string — even a line that looks like "105". You must convert it with int() or float() before doing math.
12.2 — write() and File Modes

Writing to a File

f = open("myfile.txt", "w")     # open for writing
f.write("Example string:\n")
f.write("test....")
f.close()

# write() only accepts strings --
# convert numbers first!
num1, num2 = 5, 7.5
f = open("myfile.txt", "w")
f.write(str(num1) + " + " + str(num2))
f.write(" = " + str(num1 + num2))
f.close()
ModeRead?Write?Creates?Overwrites?
"r"YesNoNoNo
"w"NoYesYesYes
"a"NoYesYesNo — appends
"w" mode erases existing content the instant the file is opened — use "a" if you want to keep what's already there.
12.2 — Behind the Scenes

Why write() Doesn't Always Write Immediately

Output Is Buffered

By default, data is line-buffered — the interpreter holds written text in memory and only flushes it to disk when it sees a newline character.

  • There can be a short delay between calling write() and the data actually landing on disk
  • f.flush() forces the buffer to disk immediately
  • f.close() also flushes automatically — another reason to always close (or use with, coming up in 12.6)
import os

f = open("myfile.txt", "w")
f.write("Write me, please!")  # not on disk yet

f.flush()          # force the buffer out
os.fsync(f.fileno())  # some OSes need this too
# now guaranteed to be on disk

12.3 — The os Module

Talking to the Operating System

import os

my_file = open("myfile.txt", "r")
file_info = os.stat("myfile.txt")   # size, access time...
my_file.close()

os.remove("myfile.txt")             # delete from disk
Why It Matters

The operating system (Windows, macOS, Linux) actually controls the file system. The os module is Python's interface to those operating-system function calls.

  • os.stat(path) — returns metadata about a file
  • os.remove(path) — deletes a file from disk (permanent!)
12.3 — os.path.join()

Write Portable Paths, Not Hardcoded Separators

The Problem

Windows uses "\\" between directories; macOS/Linux use "/". Hardcoding either one breaks your program on the other operating system.

  • os.path.join("logs", "2026", "log.txt") uses the correct separator automatically
  • os.path.sep stores the current OS's separator character
  • "path".split(os.path.sep) does the reverse — splits a path into tokens
Avoid path = "subdir\\file.txt" — instead write path = os.path.join("subdir", "file.txt"). 
import os

path = os.path.join("logs", "2026", "log.txt")
print(path)
# Windows: logs\2026\log.txt
# Mac/Linux: logs/2026/log.txt

f = open(path, "r")
print(f.read())
f.close()
12.3 — More os.path Tools

Checking, Measuring, and Walking Directories

import os

p = os.path.join("logs", "2026", "log.txt")

if os.path.exists(p):
    print("Found it.")

if os.path.isfile(p):
    print(f"Size: {os.path.getsize(p)} bytes")

print(os.path.split(p))
# ('logs/2026', 'log.txt')

for dirname, subdirs, files in os.walk("logs"):
    print(dirname, subdirs, files)
  • os.path.exists(path) — True/False if the path exists at all
  • os.path.isfile(path) — is it specifically a file (not a directory)?
  • os.path.getsize(path) — size in bytes
  • os.path.split(path) — splits into (head, tail)
  • os.walk(path) — visits every subdirectory, yielding (dirname, subdirs, files) each time
12.4 — Beyond Text

Binary Data: When a File Isn't Text

bytes Object

Images, videos, and PDFs are stored as raw binary data — not encoded text. A bytes object represents that sequence of raw byte values.

  • Add "b" to the mode to open in binary: open("img.bmp", "rb")
  • In binary mode, read() returns a bytes object, not a string
  • A bytes literal looks like b"hello"
  • Binary mode skips Windows' automatic \n → \r\n translation — important so binary data isn't corrupted
f = open("ball.bmp", "rb")   # binary mode
contents = f.read()
f.close()

print(type(contents))
# <class 'bytes'>

print(contents[:4])
# b'BMb\xe6'   -- raw header bytes
12.5 — sys.argv

Letting the User Name the File at Runtime

import sys, os

if len(sys.argv) != 2:
    print(f"Usage: {sys.argv[0]} input_file")
    sys.exit(1)

if not os.path.exists(sys.argv[1]):
    print("File does not exist.")
    sys.exit(1)

f = open(sys.argv[1], "r")
print(f.read())
f.close()

# > python my_script.py myfile1.txt
  • sys.argv is a list of strings typed after the script name on the command line
  • sys.argv[0] is always the script's own name; sys.argv[1] is the first real argument
  • sys.exit(1) ends the program and signals an error to the operating system
Always check os.path.exists() before opening a user-supplied path — don't assume it's there.
12.6 — Context Managers

Never Forget to Close a File Again

print("Opening myfile.txt")
with open("myfile.txt", "r+") as f:
    num1 = int(f.readline())
    num2 = int(f.readline())
    product = num1 * num2
    f.write("\n")
    f.write(str(product))
# No need to call f.close() -- closed automatically!
print("Closed myfile.txt")
Context Manager

with opens the file, binds it to a name via as, and guarantees it's closed the moment the indented block ends — even if an exception occurs inside.

  • Same idea as try / finally from Chapter 8, but built in and automatic
  • A forgotten close() can leave a file locked for other programs
Good practice: use with for essentially every file you open from here on.

12.7 — The csv Module

Reading Spreadsheet-Style Data

import csv

with open("grades.csv", "r") as csvfile:
    grades_reader = csv.reader(csvfile, delimiter=",")
    for row in grades_reader:
        print(row)

# ['name', 'hw1', 'hw2', 'midterm', 'final']
# ['Petr Little', '9', '8', '85', '78']
# ['Sam Tarley', '10', '10', '99', '100']
CSV = Comma-Separated Values

Each line is a row; commas separate fields. The csv module's reader object is iterable — each iteration returns one row as a list of strings.

  • delimiter="," is the default — can be changed if fields contain commas
  • Every field comes back as a string, just like reading regular text
12.7 — Reading Numbers & Writing Rows

From Rows of Strings to Real Calculations

import csv

with open("grades.csv", "r") as csvfile:
    reader = csv.reader(csvfile)
    next(reader)  # skip header row
    for row in reader:
        name = row[0]
        scores = [float(c) for c in row[1:]]
        print(f"{name}: {sum(scores):.1f}")

# Writing rows back out:
with open("out.csv", "w", newline="") as f:
    writer = csv.writer(f)
    writer.writerow(["100", "50", "29"])
  • Convert each field with float() or int() before doing math — same rule as any text file
  • Skip a header row with next(reader) or a simple first_row flag
  • csv.writer(csvfile) creates a writer object; writerow() writes one row, writerows() writes several
  • Use newline="" when opening a file for csv writing — it prevents extra blank lines on Windows

Applied — Tying It Back to Chapter 8

Real Programs Combine Files and Exceptions

filename = input("Enter file name: ")
try:
    with open(filename, "r") as f:
        total = 0
        for line in f:
            total += int(line)
    print(f"Total: {total}")
except FileNotFoundError:
    print(f"{filename} does not exist.")
except ValueError:
    print("File contains a non-numeric line.")
  • with still guarantees the file closes, even if int(line) raises inside the loop
  • FileNotFoundError is a more specific subclass of IOError — prefer it when you know exactly what you're checking for
  • This is the pattern you'll lean on for the rest of the course: open safely, process defensively
Every good file-processing program answers two questions: "What if the file isn't there?" and "What if the data inside is bad?"
