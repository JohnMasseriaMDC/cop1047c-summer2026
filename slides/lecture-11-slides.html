Section 10.1

Objects: data + the operations on that data

  • The world is made of materials — wood, metal, fabric — but people think in objects: chairs, drawers, ovens.
  • Programs are made of variables and functions — but programmers think in objects too. An object is a grouping of data (variables) and operations on that data (methods).
  • Mostly we care about the operations: a drawer's job is "put stuff in, take stuff out" — not "wood + metal bar."
  • Building a program as a collection of objects makes it more understandable, manageable, and correct.

Restaurant object

Data: name, phone, reviews, cuisines

Operations: set name/phone · add cuisine · add review · print all

CarOnLot object

Data: sticker price, days on lot, original purchase price

Operations: increment days on lot · decrease sticker price

Not included: todays_temperature, num_salespeople — those belong to the weather and the dealership, not to one car.

Section 10.1

Abstraction — use the knobs, ignore the wiring

  • Abstraction: interact with an object at a high level while low-level details stay hidden. Also called information hiding or encapsulation.
  • An oven gives you a food compartment and a heat knob. Nobody reaches inside to adjust the flame.
  • Objects support abstraction by exposing only certain functions and hiding everything else.
  • An abstract data type (ADT) is a data type whose creation and update are constrained to specific, well-defined operations. A class can implement an ADT.
  • Built-in objects are everywhere already: mystr = "Hello!" creates a str object bundling the value and methods like mystr.lower() and mystr.isdigit(). Same for int, list, dict…

You've been using objects all semester. my_list.append(5), name.split(), codes.get("MIA") — every dot was a method call on an object. Tonight you finally get to build your own.

Section 10.2

class — define your own type

time_class.py
class Time:
    """ Represents a time of day """
    def __init__(self):
        self.hours = 0
        self.minutes = 0

my_time = Time()      # instantiation
my_time.hours = 7
my_time.minutes = 15

print(f"{my_time.hours} hours", end=" ")
print(f"and {my_time.minutes} minutes")
7 hours and 15 minutes
  • A class defines a new type grouping data and functions. Its variables and functions are called attributes.
  • Instantiation = "calling" the class like a function: Time(). Each call creates one instance.
  • Instantiation automatically runs __init__ — the constructor, a method that sets up the new instance's initial state.
  • self automatically references the instance being created: self.hours = 0 creates the attribute hours on this instance.
  • Access attributes with the attribute reference operator . (dot notation / member operator).

Naming convention: classes use initial caps on each word — LunchMenu, CoinAmounts, PDFFileContents. Variables stay snake_case. Following this makes code instantly readable.

Section 10.2

One class, many instances

two_times.py
class Time:
    def __init__(self):
        self.hours = 0
        self.minutes = 0

time1 = Time()        # first instance
time1.hours = 7
time1.minutes = 30

time2 = Time()        # second, completely separate instance
time2.hours = 12
time2.minutes = 45

Chapter 9 flashback — the alias trap rides again: time2 = Time() makes a new object, but time2 = time1 makes an alias — two names, one object. Change time2.hours and time1.hours changes too. Same reference model as lists.

Section 10.3

Instance methods — behavior lives in the class

time_method.py
class Time:
    def __init__(self):
        self.hours = 0
        self.minutes = 0

    def print_time(self):
        print(f"Hours: {self.hours}", end=" ")
        print(f"Minutes: {self.minutes}")

time1 = Time()
time1.hours = 7
time1.minutes = 15
time1.print_time()
Hours: 7 Minutes: 15
  • A function defined inside a class is an instance method — call it with dot notation: time1.print_time().
  • self is bound to the instance automatically: when time1.print_time() runs, self is time1. You never pass it yourself.
  • Inside the method, self.hours reaches that instance's attributes and other methods.
  • __init__ is a special method — double underscores before and after signal behavior the interpreter handles specially. Avoid double underscores in your own names.
Section 10.3

The #1 rookie error: forgetting self

payroll_bug.py
class Employee:
    def __init__(self):
        self.wage = 0
        self.hours_worked = 0

    def calculate_pay():        # BUG: forgot the self parameter
        return self.wage * self.hours_worked

alice = Employee()
alice.wage = 9.25
alice.hours_worked = 35
print(f"Alice earned {alice.calculate_pay():.2f}")
TypeError: calculate_pay() takes 0 positional arguments but 1 was given
  • Every method call automatically inserts the instance as the first argument. Your parameter list must have a seat for it: def calculate_pay(self):.
  • Read that error message carefully — "takes 0 … but 1 was given" when you passed nothing is the fingerprint of a missing self. You will see this error. Now you can fix it in ten seconds.

Section 10.4

Dot lookup: instance first, then class

gmt_offset.py
class Time:
    gmt_offset = 0                    # class attribute — one shared dial

    def __init__(self):
        self.hours = 0                # instance attributes
        self.minutes = 0

    def print_time(self):
        offset_hours = self.hours + self.gmt_offset   # local variable
        print(f"Time -- {offset_hours}:{self.minutes}")

time1 = Time(); time1.hours = 10; time1.minutes = 15
time2 = Time(); time2.hours = 12; time2.minutes = 45

Time.gmt_offset = -8      # turn ONE dial -> ALL instances change behavior
time1.print_time()
time2.print_time()
Time -- 2:15 Time -- 4:45
  • Dot notation on an instance searches the instance namespace first, then the class namespace. self.hours finds instance data; self.gmt_offset falls through to the class.
  • If a class and instance attribute share a name, the instance wins. Good practice: don't create that collision.
Section 10.5

Constructors with parameters

race_time.py
class RaceTime:
    def __init__(self, start_hrs, start_mins, end_hrs, end_mins, dist):
        self.start_hrs = start_hrs      # save each argument as an
        self.start_mins = start_mins    # instance attribute so every
        self.end_hrs = end_hrs          # method can use it later
        self.end_mins = end_mins
        self.distance = dist

    def print_pace(self):
        total_minutes = (self.end_hrs - self.start_hrs) * 60 \
                        + (self.end_mins - self.start_mins)
        print(f"Avg pace (mins/mile): {total_minutes / self.distance:.2f}")

race_time = RaceTime(5, 30, 6, 24, 5.0)   # arguments -> __init__
race_time.print_pace()
Avg pace (mins/mile): 10.80
  • Add parameters after self; pass the matching arguments at instantiation: RaceTime(5, 30, 6, 24, 5.0).
  • The pattern self.distance = dist is how a constructor argument survives — attributes persist; parameters vanish when __init__ returns.

Default values — the typical case, pre-filled

Section 10.6

Derived classes: inherit, then extend

inventory.py
class Item:                       # base class
    def __init__(self):
        self.name = ""
        self.quantity = 0
    def set_name(self, nm):
        self.name = nm
    def set_quantity(self, qnty):
        self.quantity = qnty
    def display(self):
        print(self.name, self.quantity)

class Produce(Item):              # derived from Item
    def __init__(self):
        Item.__init__(self)       # call base constructor
        self.expiration = ""
    def set_expiration(self, expir):
        self.expiration = expir

item2 = Produce()
item2.set_name("Apples")          # inherited from Item!
item2.set_quantity(40)
item2.set_expiration("May 5, 2027")
item2.display()                   # inherited from Item!
Apples 40
  • Derived class = a class that inherits the class attributes (including methods) of a base class: class Produce(Item):.
  • Produce is an Item with one addition — no copy-pasting set_name, display, etc.
  • Instance attributes are not inherited automatically — call the base constructor yourself: Item.__init__(self) hands the new Produce instance to Item's __init__, which creates name and quantity on it.
  • Any class can be a base class; no changes to it are needed.

Section 11.1

Modules — write once, import everywhere

  • A script is a Python file passed to the interpreter. Close the interactive interpreter and everything is lost; scripts persist.
  • Copying the same functions into every script gets old fast. A module is a file of Python code that scripts, other modules, or the interpreter can import.
  • To import = execute the module's code and make its definitions available to the importer.
  • A module another program requires is called a dependency.
my_funcs.py  +  main.py
# my_funcs.py — the module
def factorial(num):
    x = 1
    for i in range(1, num + 1):
        x *= i
    return x

# main.py — the script
import my_funcs
n = int(input("Enter number: "))
print(my_funcs.factorial(n))
Enter number: 5 120
  • Rules: the filename must end in .py; you import it without the extension; import names are case-sensitive (import ABC is not import aBc).
  • Good practice: all imports at the top of the file — a reader sees every dependency at a glance.
Section 11.1

What actually happens on import

1  Check sys.modules — already imported? Reuse the loaded module. Done.
2  Otherwise: create a new module object (a namespace for the module's definitions) and insert it into sys.modules.
3  Execute the module's code inside that namespace, then add the module to the importer's namespace.

  • sys.modules is a dictionary of every loaded module — that's why importing the same module twice is free: step 1 short-circuits.
  • Because step 3 executes the module, any loose print() or input() in it runs at import time. Hold that thought — it becomes the whole point of Section 11.6.
  • Once imported, use dot notation: my_funcs.factorial(5). You can even rebind a module's names — HTTPServer.address = "www.yahoo.com" — but only for the current run.

Section 11.5

How Python finds a module

1  Built-in modules first — sys, time, math ship with Python.
2  Then the directories in sys.path:
   a. the executing script's own directory
   b. directories listed in the PYTHONPATH environment variable
   c. the Python installation directory

  • Simple projects: keep your module next to your script (that's rule 2a) and imports just work.
  • An environment variable lives in the operating system, not your script — every program can read it. Windows: set PYTHONPATH="c:\dir1;c:\other\directory".

Never name your file after a built-in module. Create your own math.py and... nothing breaks loudly — Python simply loads the built-in math first, and you'll spend an hour wondering why your functions "don't exist." Ask me how students discover this one.

Section 11.6

The __name__ == "__main__" guard

web_search.py
def search_term(term):
    # ... returns result URLs ...

def stringify_result(term, domains, s):
    # ... formats the results ...

if __name__ == "__main__":
    # Runs ONLY when executed as a script
    print("Welcome to Web Search")
    term = input("Enter search term: ")
    print(search_term(term))
  • Importing a module executes all of its top-level code — including prompts and prints. That's an unwanted side effect when another script just wants your functions.
  • __name__ is a string variable automatically added to every module: normally the module's name ("web_search")…
  • …but for the executing script it is always "__main__".
  • So the guard means: "run this user interface only when I'm the program — stay quiet when I'm imported."

This idiom is everywhere. Open almost any real Python file on GitHub and you'll find it at the bottom. After tonight, your files should have it too — including your DNA project.

Section 11.7

Reloading a module without restarting

long_running.py
from importlib import reload
import send_gmail

# ... send_gmail.py's source code gets edited while we run ...

reload(send_gmail)          # re-executes the module IN PLACE
send_gmail.send(...)        # now uses the updated definitions
  • Modules execute once at import — editing the source afterward changes nothing in the running program.
  • reload() (from the importlib standard library module) re-executes the module and updates its namespace in place.
  • Useful for long-running programs — a web server updating one component without disconnecting every client.

Gotcha: names imported with from send_gmail import header are copies — reloading the module does not update them. send_gmail.header shows the new value; your bare header keeps the old one.

Chapters 10 & 11 · Recap

What you can do now

  • Objects — data + operations, behind an abstraction. You've used them all semester; now you build them.
  • Classes — class Name: defines a type; __init__(self) constructs each instance; dot notation reads and writes attributes.
  • Methods — functions in a class; self is the instance, auto-inserted as the first argument. "takes 0 positional arguments but 1 was given" = forgot self.
  • Class vs instance attributes — shared constant vs per-object state; lookups check the instance namespace first, then the class.
  • Constructors — required parameters first and in order; defaults with name=value; save arguments as self.x so they persist.
  • Inheritance — class Derived(Base): inherits class attributes and methods; call Base.__init__(self) to get the base's instance attributes.
  • Modules — import mod vs from mod import name; imports execute the module once, cached in sys.modules.
  • Packages — directories with __init__.py; dotted paths organize big projects.
  • The guard — if __name__ == "__main__": keeps script behavior out of importers; importlib.reload() refreshes a module in a running program.
